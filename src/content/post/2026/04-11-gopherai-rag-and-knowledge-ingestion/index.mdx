# 第三篇：从 0 到 1 搭建 GopherAI 的文件上传、知识入库与 RAG 链路

## 一、本篇要解决什么问题

聊天链路跑通以后，下一个非常自然的问题就是：

- 用户上传的文件到底怎么进入系统？
- 为什么“文件上传成功”不等于“知识库可用”？
- RAG 检索到底依赖什么元数据？
- 为什么不能只靠 `uploads/<username>` 目录扫文件？
- 多用户场景下，知识隔离应该放在哪一层做？

很多项目在这里会先做一个“能传文件”的接口，然后把文件名塞进向量库，表面上功能是跑通了，但很快就会遇到下面这些问题：

1. 文件名变了，索引还找得到吗？
2. 服务重启后，只靠扫目录能知道哪份文件是当前生效的吗？
3. 索引创建失败时，接口应该算成功还是失败？
4. 替换知识文件时，旧索引、旧文件、旧元数据怎么一起收口？
5. 以后如果想从“单文件”演进到“多知识库”，现在的设计还能不能撑住？

所以这一篇的目标不是“做一个上传按钮”，而是从 0 到 1 把 GopherAI 的知识入库链路搭成一个真正可维护的工程模块。

这一轮在项目里的核心改造包括：

- 新增 `model/knowledge_file.go`，把知识文件提升成显式元数据实体
- 新增 `dao/knowledgefile/knowledgefile.go`，统一读写知识文件元数据
- 新增 `internal/modules/knowledge/service.go`，承接上传、保存、索引、入库、清理的主流程
- 让 `service/file/file.go` 退化成兼容包装层
- 更新 `common/rag/rag.go`，让检索优先依赖元数据而不是扫目录
- 更新 `common/redis/key.go` 和 `common/redis/redis.go`，让向量索引改用稳定 `indexKey`
- 更新 `common/mysql/mysql.go`，把 `knowledge_files` 纳入自动迁移
- 更新 `utils/utils.go`，补上服务端文件后缀和大小校验

这一步的本质，是把“文件上传”提升成“知识入库模块”。

## 二、先看整体流程：文件是怎么变成可检索知识的

### 1. 上传与入库主流程

```text
前端选择 .md / .txt 文件
-> POST /file/upload
-> Controller 从表单取 file，并从鉴权上下文取当前用户
-> Knowledge 模块先做服务端校验：扩展名、空文件、大小上限
-> 读取该用户当前生效的知识文件元数据
-> 生成新的 fileID、storedName、indexKey
-> 把文件保存到 uploads/<username>/<fileID>.<ext>
-> 创建 Redis 向量索引
-> 调用 indexer 读取文件内容并写入向量库
-> 写 knowledge_files 元数据表
-> 清理旧文件、旧索引、旧元数据
-> 返回上传成功
```

这条链路里最重要的动作，不是“把文件写进磁盘”，而是把“文件本体、索引状态、元数据状态”三件事绑定起来。只有三者都完成了，我们才能说这份知识真正可用。

### 2. 查询与检索主流程

```text
用户在聊天中选择 RAG 模型并发起提问
-> 模型层调用 rag.NewRAGQuery(ctx, username)
-> RAG 模块先查 knowledge_files，找到当前生效的 indexKey
-> 如果查不到元数据，再兼容性地回退到 uploads/<username> 扫目录
-> 基于 indexKey 创建 Redis Retriever
-> 用 query 生成向量检索相关文档
-> 把召回文档拼进 Prompt
-> 再交给大模型生成最终回答
```

这一步体现出一个非常关键的工程认知：

**上传阶段解决的是“知识如何入库”，检索阶段解决的是“知识如何被找到”，这两个阶段不能混成一件事。**

### 3. 失败回滚流程

```text
文件保存成功
-> 创建索引或写入索引失败
-> 删除刚落盘的新文件
-> 删除刚创建的新索引
-> 不写入 knowledge_files 元数据
-> 整个上传请求返回失败
```

这也是为什么这次重构后，我们不把“文件进了磁盘”当成成功标准，而是把“文件、索引、元数据”三者一致当成成功标准。

### 4. 兼容旧数据流程

```text
检索时先查 knowledge_files
-> 如果查到元数据，就按 metadata.index_key 走新链路
-> 如果没查到，就回退到旧目录扫描逻辑
-> 保证历史上传过的文件仍然能被检索
```

这一步是为了让项目能平滑演进，而不是每次重构都把旧数据一次性打断。

## 三、项目里该怎么实现这部分能力

### 1. 为什么知识文件必须单独建模

如果项目里只有磁盘文件，没有元数据表，系统会遇到几个现实问题：

- 不知道哪份文件才是“当前生效”的知识文件
- 文件名一变，索引定位就会变脆弱
- 后面很难记录原始文件名、存储文件名、索引键、大小、创建时间
- 很难实现替换、归档、删除、版本切换、多知识库等能力

因此这次新增了 `KnowledgeFile` 模型：

```go
type KnowledgeFile struct {
    ID           string
    UserName     string
    OriginalName string
    StoredName   string
    StoragePath  string
    IndexKey     string
    SizeBytes    int64
    CreatedAt    time.Time
    UpdatedAt    time.Time
    DeletedAt    gorm.DeletedAt
}
```

这些字段分别承载不同职责：

- `ID`：知识文件唯一身份
- `UserName`：知识文件归属用户
- `OriginalName`：原始上传文件名，便于展示和审计
- `StoredName`：实际落盘名，避免直接依赖用户原始文件名
- `StoragePath`：物理存储位置
- `IndexKey`：向量索引键
- `SizeBytes`：大小信息，便于校验和后续运营约束
- `DeletedAt`：软删除标记，便于替换和审计

### 2. 为什么上传主链路要抽到 `internal/modules/knowledge`

这次没有把“校验文件、保存文件、建索引、写元数据、删旧文件”继续散落在 `controller/service/common/rag` 里，而是专门建立了知识模块边界：

- `controller/file/file.go` 负责 HTTP 入参和响应
- `service/file/file.go` 只保留兼容包装
- `internal/modules/knowledge/service.go` 负责上传主流程
- `dao/knowledgefile/knowledgefile.go` 负责元数据读写
- `common/rag/rag.go` 负责索引与检索
- `common/redis/*` 负责 Redis 索引命名和生命周期

这样做的好处是，后面无论继续加：

- 多文件知识库
- 文档状态机
- 异步切片与分块
- 解析 PDF / Word / HTML
- 重建索引任务队列
- 知识版本切换

都可以沿着 `knowledge` 模块边界继续长，而不是再回到“到处找上传相关代码”的状态。

### 3. 上传链路里每一步分别做了什么

当前 `UploadRAGFile` 的核心流程可以概括成下面八步：

1. `ValidateFile` 做服务端校验
2. 根据用户构造知识目录 `uploads/<username>`
3. 查询用户当前最新一份知识文件元数据
4. 生成 `fileID`、`storedName`、`indexKey`
5. 把上传文件保存到磁盘
6. 创建 Redis 索引并把文档内容写入向量库
7. 写入 `knowledge_files` 元数据表
8. 清理被替换的旧文件、旧索引和旧元数据

这里有两个很关键的实现细节。

第一个细节是，落盘名不再直接使用用户原始文件名，而是：

```go
storedName := fileID + strings.ToLower(filepath.Ext(originalName))
```

这样做的价值有三层：

- 避免用户文件名冲突
- 避免后续重命名影响索引定位
- 降低路径处理和原始文件名污染的风险

第二个细节是，索引键不再直接使用文件名，而是绑定到稳定 `fileID`：

```go
indexKey := buildIndexKey(fileID)
```

这意味着向量索引的身份不再依赖“文件叫什么”，而依赖“这份知识文件是谁”。

### 4. 为什么索引要用稳定 `indexKey`，而不是直接用文件名

这是这次重构里非常关键的一点。

原来的问题是：

- 文件名天然不稳定
- 同名覆盖和重传会让索引语义混乱
- 目录扫描拿到的文件名，本质上不是强约束元数据
- 文件名一旦被改造，向量索引定位就会变脆

现在改成 `indexKey` 以后，规则就清晰了：

1. 每次新知识文件生成一个新 `fileID`
2. `indexKey` 绑定这次上传
3. Redis 索引名、前缀、文档键都基于 `indexKey`
4. 检索时先通过元数据找到 `indexKey`，再去查向量库

这其实是在做一件很工程的事：

**把“索引身份”从文件系统偶然状态，提升成持久化元数据。**

### 5. 检索链路为什么要先查元数据，再做目录兼容回退

`rag.NewRAGQuery(ctx, username)` 现在会优先走新链路：

- 先查 `knowledge_files`
- 查到则直接拿 `IndexKey`
- 查不到才回退到 `uploads/<username>` 扫目录

这样做有两层含义：

第一层是新架构的事实来源已经变成元数据表，而不是目录。

第二层是为了平滑兼容旧数据。因为历史用户可能已经传过文件，但当时没有 `knowledge_files` 表。如果完全去掉扫目录逻辑，旧用户会直接失去知识检索能力。

这也说明一件事：

**真正好的重构，不是只让新逻辑更漂亮，还要尽量照顾旧数据如何过渡。**

### 6. 为什么这次只保留“一用户一份当前生效知识文件”

当前版本仍然保留了较轻量的业务约束：

- 一个用户当前只有一份生效知识文件
- 上传新文件时，会清理旧文件、旧索引和旧元数据

原因不是这个模型最强，而是它最适合当前阶段：

- 前端交互还比较轻
- 聊天侧只需要“有没有一份当前知识库”
- 先把上传、替换、查询、回滚、兼容这些边界打稳更重要

面向后续演进时，这个结构也留了口子，因为：

- 我们已经有了独立元数据表
- 已经有显式 `IndexKey`
- 已经把上传主链路从控制器里抽出来

所以以后从“一用户一份生效文件”演进到“多知识库、多版本、多状态”，不是推翻重来，而是在当前模型上继续往里长。

## 四、为什么这次要这样重构，而不是继续沿着旧逻辑补丁式修改

### 1. 因为文件系统不是事实来源

文件系统适合存放文件本体，但它不适合承担业务事实来源的角色。

如果只靠扫目录，你很难回答这些问题：

- 哪个文件是当前正在使用的？
- 这个文件对应哪个向量索引？
- 这个文件什么时候上传的？
- 替换过几次？
- 是否已经被逻辑删除？

这些都更适合放在数据库元数据里。

### 2. 因为“上传成功”不等于“知识可用”

这是非常值得在项目和面试里讲清楚的一点。

如果上传接口只是把文件写进磁盘就返回成功，那么可能出现：

- 文件已经存在
- 但向量索引没建成功
- 或者索引建了，但元数据没写成功
- 聊天侧最终仍然找不到这份知识

所以这里真正的成功标准应该是：

**文件成功保存 + 索引成功建立 + 元数据成功持久化。**

缺任何一环，都不能算“知识入库完成”。

### 3. 因为入库阶段和检索阶段本来就不是一回事

很多人初学 RAG 时，容易把它理解成一句话：

“把文件丢进向量库，然后检索一下。”

这句话不能说错，但它隐藏了至少两条不同链路：

- 入库链路：解析、切片、Embedding、存储、记录元数据
- 检索链路：定位索引、召回文档、拼 Prompt、生成回答

把这两条链路拆清楚，后面你才容易继续做：

- 分块策略优化
- 混合检索
- 重排
- Query Rewrite
- 多知识库路由

### 4. 因为兼容旧数据本身就是工程能力

很多重构失败，不是因为新逻辑写得差，而是因为它默认“历史系统不存在”。

这次保留目录回退逻辑，说明我们在设计时考虑的是：

- 新结构怎么上线
- 历史用户怎么过渡
- 改造期间业务不能直接断

这类思考在国内大中厂的项目面试里非常加分，因为它体现的不是“我会写新代码”，而是“我知道怎么在真实业务里替换旧链路”。

## 五、实现中必须特别注意的坑

### 1. 前端 `accept` 不是安全校验，服务端必须自己校验

前端限制用户只选 `.md`、`.txt`，只是交互辅助，不是安全边界。服务端仍然必须检查：

- 文件扩展名
- 文件大小
- 空文件
- 后续还应该补 MIME / 内容嗅探

也就是说，真正可信的校验永远在服务端。

### 2. 不能直接把用户原始文件名当成真实存储名

原因包括：

- 同名覆盖风险
- 特殊字符和路径处理风险
- 原始文件名改变后不利于稳定定位
- 安全审计和清理逻辑更容易混乱

所以原始文件名适合做展示字段，不适合直接做系统内部关键标识。

### 3. 半成功状态一定要回滚

当前链路里，一旦“保存文件之后、写元数据之前”的步骤失败，就会主动清理新文件和新索引。

这件事非常重要，因为没有回滚的话，系统里就会出现：

- 磁盘上有孤儿文件
- Redis 里有孤儿索引
- 数据库里却没有可追踪元数据

这类脏数据后面会越来越难排查。

### 4. 软删除比物理删除更适合替换型业务

这次元数据删除走的是软删除。原因是替换知识文件这类动作，往往还需要保留一定审计能力：

- 谁上传过什么
- 什么时候被替换
- 什么时候被清理

而且后面如果要补“恢复”“版本对比”“操作记录”，软删除也更容易接。

### 5. RAG 的索引模型和查询模型最好保持一致

当前索引和查询都使用同一个 Embedding 配置。这是因为向量检索的核心前提就是：

- 文档向量
- 查询向量

必须处在同一个语义空间里。

如果索引和查询用了不一致的 Embedding 模型，召回质量通常会明显变差，甚至完全不可用。

### 6. 文件上传只是 RAG 的入口，不是 RAG 的全部

当前实现仍然比较轻量：

- 整个文件作为单文档入库
- 还没有做 chunking
- 还没有做 rerank
- 还没有做 query rewrite
- 还没有做多路召回

所以这一轮解决的是“链路正确性”和“工程边界”，不是“RAG 效果已经做到最优”。

### 7. 多用户隔离要在多个层面同时考虑

至少包括：

- 文件目录按用户隔离
- 元数据按用户隔离
- 检索时按用户定位生效索引
- 聊天请求仍然要基于登录态拿当前用户

只做其中一层，不足以构成完整隔离。

## 六、围绕本篇内容，需要进一步掌握哪些技术基础

### 1. `multipart.FileHeader` 到底是什么

Gin 上传接口拿到的并不是文件内容本身，而是 `*multipart.FileHeader`。它包含：

- 文件名
- 大小
- 表单字段信息

真正读取文件内容时，还要调用 `file.Open()` 拿到流，再复制到目标文件里。

### 2. RAG 最基础的四步是什么

如果先用工程视角做最小抽象，RAG 至少包含：

1. 把文档转成可处理文本
2. 把文本变成向量并入库
3. 用用户问题做相似检索
4. 把召回文档拼进 Prompt 再交给大模型回答

当前这轮重构主要做的是第 2 步和第 3 步之间的工程衔接，让它们能被稳定追踪和定位。

### 3. Redis 在这里承担的是什么角色

当前 Redis 不是在这里当缓存，而是在配合 RediSearch 做向量索引与检索。也就是说：

- 它保存文档向量
- 保存文档内容与元数据
- 接收相似度检索请求

所以你在面试里不要只说“Redis 用来缓存”，在这个项目场景里它更接近“轻量向量检索底座”。

### 4. 为什么数据库元数据和向量库必须分工

向量库擅长回答的问题是：

- 哪些文本和 query 最相似？

数据库擅长回答的问题是：

- 这份知识属于谁？
- 当前生效的是哪份？
- 文件什么时候上传的？
- 是否已被替换或删除？

所以不要把“向量库能存一些字段”误解成“它就能替代业务数据库”。

### 5. 为什么安全上传是一个独立知识点

文件上传表面上很简单，但它背后至少牵扯：

- 路径处理
- 类型校验
- 大小控制
- 资源滥用
- 恶意文件
- 存储隔离
- 回滚清理

所以真正的文件上传，不只是“接口能收到文件”，而是“系统能安全、可控地接收文件”。

## 七、推荐学习资源

下面这些资源尽量按“官方 / 权威优先”整理，适合围绕本篇继续深入。

- [Go Packages: mime/multipart](https://pkg.go.dev/mime/multipart)

- [Go Packages: context](https://pkg.go.dev/context)
  作用：理解超时、取消、请求级上下文传播。
- [Go Blog: Context](https://go.dev/blog/context)
  作用：理解 Go 官方推荐的 `context` 使用方式，以及为什么请求级任务要把取消信号一路往下传。

### 4. GORM 模型与软删除

  作用：理解模型字段、时间字段、主键和软删除的约定。

  作用：理解为什么替换型业务里常会优先使用软删除，而不是直接物理删除。

## 八、提问 + 标准答案

下面这些问题既适合你做项目复盘，也非常贴近国内大中厂围绕 RAG、上传链路和工程化的常见追问。

### 问题 1：什么是 RAG？它和微调有什么区别？

**标准答案：**

RAG 是 Retrieval-Augmented Generation，也就是“检索增强生成”。它先从外部知识中召回相关内容，再把这些内容拼进 Prompt 交给大模型作答。微调是把知识或风格固化进模型参数里。RAG 更适合知识经常变化、希望快速更新的场景；微调更适合稳定能力或风格迁移。这个项目当前选 RAG，是因为文档知识更新比重新训练模型更轻量。

### 问题 2：为什么不能只把文件传到服务器上，就算知识库完成了？

**标准答案：**

因为上传文件只完成了“文件本体保存”，还没有保证“索引可检索”和“元数据可定位”。如果文件已经落盘，但向量索引没建好，聊天链路仍然无法正确检索这份知识。所以知识库完成的标准应该是文件、索引、元数据三者都成功。

### 问题 3：为什么要单独建 `knowledge_files` 表？

**标准答案：**

因为文件系统只适合存文件，不适合存业务事实。`knowledge_files` 表能显式记录用户归属、原始文件名、真实存储名、索引键、大小、创建时间和软删除状态。后面做替换、审计、版本、归档、多知识库时，这些元数据都必不可少。

### 问题 4：为什么向量索引不能直接用文件名当 key？

**标准答案：**

因为文件名不稳定，容易受重传、重命名、同名覆盖和目录扫描逻辑影响。把索引键绑定到稳定 `fileID` 上，本质上是把“索引身份”从文件系统状态提升为业务元数据，这样定位更稳、清理更稳、演进也更稳。

### 问题 5：为什么说文件系统不是事实来源？

**标准答案：**

因为文件系统只能告诉你“这里有个文件”，但回答不了“这是不是当前生效文件”“对应哪个索引”“是否已逻辑删除”“什么时候上传的”“替换过几次”等业务问题。这些都应该由数据库元数据承担。

### 问题 6：为什么这次把上传主链路抽到了 `internal/modules/knowledge`？

**标准答案：**

因为上传、索引、元数据、清理已经形成了一个完整业务闭环，不再是单一控制器动作。抽成模块以后，后面不管做多文件、多状态、异步解析还是文档版本，都会有更稳定的扩展边界。

### 问题 7：为什么服务端还要做文件校验，前端不是已经 `accept=.md,.txt` 了吗？

**标准答案：**

前端校验只是交互辅助，不是安全边界。真正可信的边界永远在服务端。用户完全可以绕过前端直接构造请求，所以后端必须自己校验后缀、大小、空文件，后续还应继续补 MIME 和内容校验。

### 问题 8：为什么这次先限制成一用户一份生效知识文件？

**标准答案：**

因为当前阶段更重要的是先把上传、索引、替换、清理、检索和兼容旧数据的主链路打稳。单活知识文件能降低前后端复杂度，也更容易验证 RAG 主链路是通的。等元数据和索引边界稳定后，再扩成多知识库会更自然。

### 问题 9：为什么替换文件时要同时清理旧文件、旧索引和旧元数据？

**标准答案：**

因为这三者代表的是同一份知识的不同投影。如果只删其中一部分，就会留下脏数据，比如磁盘孤儿文件、Redis 孤儿索引或数据库无效元数据。时间一长，排查问题会越来越困难。

### 问题 10：为什么元数据删除更适合先做软删除？

**标准答案：**

因为替换类业务往往还需要审计和追溯能力。软删除可以保留历史记录，后面无论做恢复、操作日志还是版本分析，空间都会更大。

### 问题 11：RAG 检索时为什么要先查元数据，再查向量库？

**标准答案：**

因为向量库擅长做相似检索，但不擅长回答“当前该查哪个索引”。这个问题属于业务元数据范畴，应该先由数据库决定生效索引，再进入向量检索。

### 问题 12：为什么索引阶段和查询阶段最好使用同一个 Embedding 模型？

**标准答案：**

因为文档向量和查询向量必须位于同一个语义空间里，检索相似度才有意义。如果索引时和查询时使用不同 Embedding 模型，召回质量通常会明显下降。

### 问题 13：Redis 在这个项目里只是缓存吗？

**标准答案：**

不是。当前这个场景里 Redis 主要承担的是向量检索底座的角色，配合 RediSearch 建索引、存向量、做相似召回。也就是说，它在这里更像轻量向量数据库能力，而不是传统意义上的 KV 缓存。

### 问题 14：为什么说“入库”和“检索”应该拆成两个阶段来讲？

**标准答案：**

因为两者的目标完全不同。入库阶段关注的是文件解析、向量生成、索引构建和元数据登记；检索阶段关注的是定位正确索引、召回正确上下文、拼 Prompt 并生成回答。拆开以后，链路定位、故障排查和后续优化都会更清晰。

### 问题 15：为什么这轮要保留旧目录扫描的兼容逻辑？

**标准答案：**

因为真实业务里不能假设所有旧数据都能瞬间迁移完成。保留回退逻辑能让历史上传过文件的用户在重构后仍然可用，这体现的是渐进式迁移能力，而不是只顾新代码整洁。

### 问题 16：文件上传链路里最值得警惕的安全问题有哪些？

**标准答案：**

常见风险包括恶意文件上传、伪装后缀、超大文件消耗资源、路径穿越、同名覆盖、存储污染和脏数据堆积。所以真实工程里需要同时做类型限制、大小限制、命名隔离、目录隔离和失败回滚。

### 问题 17：为什么聊天侧仍然要依赖登录态来决定检索哪个知识库？

**标准答案：**

因为知识库是用户维度资源，不是公共资源。即便向量库里已经有很多索引，聊天请求也必须先根据登录用户找到属于他的那份生效索引，否则就会出现跨用户知识串用问题。

### 问题 18：如果面试官问你“RAG 效果不好时你先优化什么”，你怎么回答？

**标准答案：**

我会按链路拆开排查：先看文档是否被正确解析和分块，再看 Embedding 模型是否合适，然后看召回 `TopK`、向量索引、过滤条件，最后再看是否需要 Query Rewrite、混合检索或 Rerank。也就是说，不会一上来就盲目换模型，而是先判断问题出在入库、召回还是生成阶段。

### 问题 19：如果以后要支持多知识库，你会怎么扩展当前设计？

**标准答案：**

我会保留当前 `knowledge_files` 元数据表，但增加状态字段、知识库分组、版本或标签维度，并让聊天请求能够显式选择知识库集合。同时把“单活文件清理”改成“按知识库或版本切换”，而不是全量替换。因为当前设计已经把元数据和索引键抽出来了，扩展空间是存在的。

### 问题 20：如果让你继续优化这块链路，下一步你会补什么？

**标准答案：**

我会优先补：

- 文档分块与 chunk 元数据
- 异步入库任务和状态机
- PDF / DOCX / HTML 解析
- 更严格的 MIME 与内容校验
- 检索质量评估与召回指标
- 多知识库路由与版本管理
- 更完整的索引清理和幂等机制

这说明项目会从“能上传并检索文件”继续演进到“能稳定、可观测地管理知识系统”。
