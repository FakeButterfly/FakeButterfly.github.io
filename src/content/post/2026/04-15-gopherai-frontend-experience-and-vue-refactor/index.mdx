## 一、本篇要解决什么问题

当 GopherAI 逐步拥有认证、聊天、RAG、MCP、多模态和可观测性之后，前端如果还停留在“页面能打开、按钮能点、接口能通”的阶段，很快就会暴露出三个问题：

- 页面职责混乱，登录、导航、工作流、请求处理都耦合在一起。
- API 调用分散，401、业务码、异常提示的处理方式不统一。
- 聊天页、上传页、图像识别页越来越像 Demo，而不是一个可以长期演进的产品前端。

所以这一阶段的目标不是单纯“换个样式”，而是把前端搭成真正可持续演进的工程骨架。

## 二、从零搭建一个 AI 应用前端时，我会先搭什么骨架

如果我们从零开始做 GopherAI 的前端，我会先搭四层：

1. 路由边界：区分公开页和受保护页面。
2. 应用外壳：给受保护页面一个统一的导航、标题和退出入口。
3. 请求边界：把 token、错误处理、业务码拆到统一 API 层。
4. 工作流页面：聊天、上传、识别这些页面都按“任务流”来设计，而不是按“控件堆砌”来设计。

这四层稳住之后，再去做视觉、交互和性能优化，才不会越改越乱。

## 三、GopherAI 前端主流程是怎样跑起来的

这一轮前端重构之后，核心流程可以概括成下面这条链路：

```text
注册页 -> 发送验证码 -> 完成注册 -> 保存 token -> 进入受保护应用
登录页 -> 输入邮箱里收到的 11 位用户名 + 密码 -> 保存 token -> 进入应用
应用外壳 -> 侧边栏导航 -> 进入 Dashboard / AI Chat / Image AI
AI Chat -> 拉取会话列表 -> 进入会话 -> 同步历史 -> 发送消息 / 流式对话 / 上传知识文件 / 发起 TTS
Image AI -> 选择图片 -> 前端校验 -> 本地预览 -> 调用识别接口 -> 展示结果与历史
```

从产品视角看，这条链路解决的是“用户如何进入系统并完成核心任务”。从工程视角看，这条链路解决的是“认证、路由、请求、任务页面如何稳定协同”。

## 四、这一阶段具体是怎么实现的

### 1. 用路由守卫和应用外壳，先划清页面边界

这一轮把前端页面分成了两类：

- 公开页：`/login`、`/register`
- 受保护页：`/menu`、`/ai-chat`、`/image-recognition`

实现上做了两件事：

1. 在 `src/router/index.js` 里增加公开路由与受保护路由的元信息，并在全局守卫里判断当前是否有 token。
2. 引入 `src/layouts/AppShell.vue` 作为受保护页面的统一外壳，把导航、标题、当前页面说明、退出登录这些共性能力收回来。

这样做的好处非常直接：

- 页面边界清晰，不再每个页面都重复写导航和退出逻辑。
- 登录后跳转、未登录拦截、标题切换都能集中处理。
- 后续增加新页面时，只需要挂进外壳，而不是重新复制一套结构。

### 2. 用统一 API 层收口 token、401 和业务码处理

在很多早期项目里，页面会直接写：

```js
axios.post('/xxx', data)
```

短期内很快，但很快就会遇到问题：

- token 从哪里拿？
- 401 要不要统一跳转登录？
- 后端 `HTTP 200` 但 `status_code != 1000` 怎么处理？
- SSE 流式返回和普通 JSON 怎么统一？

所以这一轮把请求层收到了 `src/utils/api.js`，把能力拆成几类：

- token 管理：`getToken`、`saveToken`、`clearToken`
- 普通请求：`request`、`unwrapResponse`、`toFriendlyError`
- 业务分组：`authApi`、`chatApi`、`knowledgeApi`、`imageApi`、`ttsApi`
- 流式对话：`streamChat`

这一步的关键不是“把 axios 包一层”，而是把**传输成功**和**业务成功**明确拆开。

也就是说，前端不能只看：

- 请求有没有返回 200

还必须看：

- 后端业务码是否真的代表成功
- 失败时应该给用户怎样的提示
- 401 是否需要集中清理登录态

### 3. 登录和注册页不再只是表单，而是完整认证入口

这一轮新建了 `src/components/auth/AuthShell.vue`，把认证页抽成了统一容器。登录页和注册页则分别处理各自流程：

- 登录页强调“登录用户名不是邮箱，而是注册邮件中收到的 11 位账号”。
- 注册页支持发送验证码、倒计时、注册成功后自动写入 token 并进入系统。

为什么要特别强调用户名这一点？因为这是一个非常典型的产品和工程协同问题。

如果前端不把这个信息讲清楚，后端明明没错，用户也会误以为“登录接口坏了”。很多所谓接口问题，最后其实是前端交互表达问题。

### 4. AI Chat 页按任务流重构，而不是继续堆一个大组件

聊天页是这一轮里最重要的页面。它现在承担的不只是“发一条消息”，而是一整条工作流：

- 拉取会话列表
- 切换当前会话
- 同步历史消息
- 发送普通消息
- 发送流式消息
- 在没有会话时自动创建会话
- 上传 `.md` / `.txt` 知识文件
- 发起 TTS 任务并轮询结果

这也是为什么聊天页必须是“工作流页面”。

如果继续只把它当作一个输入框和一个发送按钮，页面很快就会变成：

- 状态非常多
- 接口非常多
- UI 和数据逻辑严重耦合
- 出问题时很难定位是哪一层错了

这一轮在 `src/views/AIChat.vue` 里做的关键处理包括：

- 用独立的会话侧边栏展示会话列表。
- 为“新会话”和“当前会话发送”拆开逻辑。
- 通过 `streamChat` 解析 SSE 分片，渐进更新回复内容。
- 在渲染模型输出前做基础转义，再做 Markdown 展示，降低直接注入 HTML 的风险。
- 为 TTS 引入异步任务轮询，而不是阻塞当前聊天流程。

### 5. Image AI 页补齐了前端校验、预览和本地历史

图像识别页也不再只是“选文件然后调用接口”。这一轮补了三个很重要的用户体验层：

- 选择文件后立即预览。
- 前端先校验文件类型和大小，再决定是否发请求。
- 保存当前页面历史记录，让用户可以回看最近识别结果。

为什么前端也要做校验？因为前端校验虽然不是安全边界，但它是很好的第一层体验边界。

它的价值在于：

- 尽早告诉用户哪里不符合要求。
- 避免把明显错误的请求发给后端。
- 减少无意义的网络和服务端开销。

但这一步一定要记住：**前端校验是补充，不是替代。后端仍然必须做最终校验。**

### 6. 主题层和全局样式也要工程化

这一轮新增了 `src/styles/theme.css`，统一整理了页面底色、卡片层次、主题色、Element Plus 变量和一些全局排版样式。

这样做的意义不是“为了好看一点”，而是：

- 避免样式继续散落在各个页面。
- 给后续新页面提供稳定的视觉变量。
- 降低局部改动把全局页面打乱的概率。

## 五、为什么这一轮要这样实现

### 1. 为什么要有 App Shell，而不是每页自己写导航

因为一旦页面多起来，导航、标题、登录态、退出入口、全局提示这些能力就是应用级能力，不再属于某一个页面。把它们收口到 App Shell，是在给整个系统立边界。

### 2. 为什么路由守卫不能等同于安全边界

因为路由守卫只发生在前端，它最多做到：

- 改善页面跳转体验
- 避免未登录用户进入受保护页面
- 统一处理无 token 的跳转

但真正的安全边界一定在后端。用户完全可以绕过前端，直接请求 API。所以面试时一定要把这句话说清楚：**前端路由守卫是体验边界，不是安全边界。**

### 3. 为什么要统一做 401 处理

因为登录态失效是整个应用级问题，而不是单个页面问题。

如果每个页面都各自判断 401，会出现：

- 有的页面跳了登录，有的页面没跳
- 有的页面清了 token，有的页面没清
- 提示信息不一致

统一处理的价值是，把登录态失效这类横切问题放到横切层解决。

### 4. 为什么聊天页要同时支持普通请求和流式请求

因为 AI 场景里，很多时候流式输出体验明显更好。用户不需要等完整答案生成后才看到结果，而是可以边生成边阅读。

但工程上流式并不等于“更简单”，因为你要处理：

- 分片消息拼接
- 中途断开
- 流结束标记
- 当前消息占位
- 新会话创建后如何把 sessionId 回填回来

所以这一轮单独把 `streamChat` 设计成请求层能力，而不是把这套解析逻辑散落在页面里。

### 5. 为什么做了路由懒加载，打包还是会有 vendor 大包告警

因为懒加载能拆的是**页面代码**，但像 Vue、Element Plus、Markdown 相关依赖、axios 这类基础依赖，仍然可能被聚到较大的 vendor chunk 中。

所以看到构建告警时，不要立刻得出“懒加载没用”的结论。更准确的理解是：

- 路由懒加载解决的是页面按需加载。
- vendor 大包反映的是公共依赖体积仍然偏大。

这说明优化已经开始生效，但还没有走到更细的包体治理阶段。

## 六、实现时需要特别注意哪些地方

### 1. 不要把 token 放到 URL 里

更推荐放在 `Authorization: Bearer <token>` 请求头里。把 token 放进 URL，会更容易进入浏览器历史、日志、代理记录和监控系统。

### 2. 401 重定向要避免循环跳转

如果登录页本身也被错误拦截，或者公开页和受保护页边界没划清，很容易出现“登录页也在不停跳登录页”的循环问题。

### 3. 处理 SSE 时不能假设一个 chunk 就是一条完整消息

流式输出经常会出现半条数据、半个 JSON、多个 event 拼在一起的情况。所以前端一定要按协议累积缓冲区，再按行解析，而不是每次拿到 chunk 就直接当完整消息处理。

### 4. 渲染模型输出时要有 XSS 意识

如果直接把模型生成内容当 HTML 插进页面，风险很高。比较稳妥的做法是：

- 默认按文本处理
- 做基础转义
- 只在明确受控的范围内做 Markdown 渲染
- 真要支持更复杂的富文本，再额外引入消毒方案

### 5. 文件预览和定时器都要记得回收

像图片预览使用的 `Object URL`、验证码倒计时、TTS 轮询这类资源，如果不及时清理，很容易形成页面切换后的内存和状态残留问题。

### 6. 前端校验不能替代后端校验

前端只负责“尽早反馈”，后端才负责“最终兜底”。这是文件上传、认证参数校验、内容安全校验里都必须记住的原则。

## 八、提问 + 标准答案

### 问题 1：为什么一个 AI 应用的前端也要按模块分层，而不是几个页面直接写完？

**标准答案：**
因为前端一旦承接认证、聊天、上传、流式输出、多模态和监控入口，它就不再只是“几个页面”，而是完整产品前台。模块分层可以把路由、应用外壳、请求层、页面工作流和样式变量拆开，降低耦合度，方便后续继续演进。

### 问题 2：为什么登录态更推荐放在 `Authorization` 请求头里，而不是 URL 参数里？

**标准答案：**
放在请求头里更符合 HTTP 语义，也更安全。URL 更容易进入浏览器历史、反向代理日志、服务端访问日志和分享链路，泄漏风险更高。

### 问题 3：前端路由守卫能不能算真正的权限控制？

**标准答案：**
不能。路由守卫只能改善前端体验，真正的权限控制一定在后端接口。用户完全可以绕过前端页面，直接调用后端 API。

### 问题 4：为什么前端还要统一封装 axios，请求直接写在页面里不行吗？

**标准答案：**
小页面可以直接写，但项目一大就会出现 token 注入、401 处理、错误提示、业务码处理、重复逻辑分散的问题。统一请求层是为了把横切逻辑收口，让页面只关心自己的业务流程。

### 问题 5：为什么要区分“HTTP 成功”和“业务成功”？

**标准答案：**
因为很多后端接口会返回 `HTTP 200`，但业务码仍然表示失败。前端如果只看 HTTP 状态，会把很多真实失败误判成成功，进而造成提示错误、状态错误和埋点失真。

### 问题 6：为什么聊天页不能只做成一个输入框加发送按钮？

**标准答案：**
因为真实聊天页不仅有发送消息，还会涉及会话列表、历史同步、流式输出、模型切换、文件上传、TTS、错误恢复等工作流。如果继续按 Demo 思路写，组件会迅速失控。

### 问题 7：前端如何安全地消费 SSE 或流式输出？

**标准答案：**
要按协议逐块读取流，维护缓冲区，再按行解析 `event` 和 `data`。不能假设一个 chunk 就是一条完整消息，还要考虑断流、结束标记和中途报错的处理。

### 问题 8：为什么前端也要做上传校验，后端不是已经会校验了吗？

**标准答案：**
前端校验的价值在于尽早反馈、减少无效请求和提升体验；后端校验的价值在于真正保证安全和数据正确性。两者不是替代关系，而是前后协作。

### 问题 9：如果面试官问“如何避免模型输出导致 XSS”，应该怎么答？

**标准答案：**
核心原则是默认把模型输出当不可信内容处理。前端应优先按文本显示，必要时先转义，再做受控范围内的 Markdown 渲染；更复杂的富文本渲染需要进一步做 HTML 消毒。后端如果会存储和分发这些内容，也要同步做安全治理。

### 问题 10：做了路由懒加载，为什么构建时还是可能看到大包告警？

**标准答案：**
因为懒加载主要拆的是页面代码，而大型公共依赖仍可能被打进 vendor chunk。出现告警说明还需要进一步做依赖裁剪、按需引入、渲染方案优化等工作，但不代表懒加载没有价值。

### 问题 11：Vue CLI 和 Vite 在工程化上怎么选？

**标准答案：**
Vue CLI 生态成熟，适合承接已有项目；Vite 冷启动更快、开发体验更轻，适合新项目或逐步迁移。回答时不要绝对化，应结合团队存量项目、插件依赖和迁移成本来讲。

### 问题 12：为什么要给受保护页面加统一的 App Shell？

**标准答案：**
因为导航、标题、登录态和全局操作本质上属于应用级能力。统一 App Shell 可以减少重复布局、统一用户体验，并为后续页面扩展提供稳定外壳。

### 问题 13：为什么 401 要集中处理，而不是每个页面自己判断？

**标准答案：**
集中处理能保证行为一致，比如统一清 token、统一跳登录、统一提示。否则每个页面各写一套逻辑，最终会造成行为不一致和维护成本飙升。

### 问题 14：前端状态管理什么时候应该引入 Pinia 或更完整的 store？

**标准答案：**
当状态跨多个页面共享、更新来源复杂、需要统一缓存和派生状态时，就应该考虑引入集中状态管理。如果状态仍然主要局限在单页工作流里，先用页面内状态和明确的请求层往往更简单。

### 问题 15：如果大厂面试官问“前端如何配合可观测性”，应该怎么答？

**标准答案：**
前端至少要保证请求 ID 透传、错误提示一致、关键用户操作可追踪、异常链路可还原。更进一步可以做埋点、性能指标采集、前端错误上报，并和后端 request id / trace id 串联起来。

### 问题 16：为什么说前端性能优化不能只盯着 Lighthouse 分数？

**标准答案：**
因为真实用户体验不仅包括首屏指标，还包括接口等待、交互延迟、流式反馈速度、失败恢复体验和页面稳定性。AI 应用里，流式响应感知往往比单纯静态分更重要。

### 问题 17：本阶段前端重构最重要的工程收益是什么？

**标准答案：**
不是“页面更好看了”，而是边界更清楚了。登录态、路由、请求层、聊天工作流、文件上传和图像识别的职责开始稳定下来，后续继续加 RAG、MCP 和更多前端入口时，复杂度不会再线性失控。
