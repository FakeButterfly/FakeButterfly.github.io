# 第二篇：从 0 到 1 搭建 GopherAI 的会话与聊天模块

## 一、本篇要解决什么问题

认证层和启动层搭好以后，真正进入 AI 应用核心的，就是会话与聊天链路。

很多人第一次做聊天功能时，容易把它理解成一句话：

“前端把问题传给后端，后端调一次模型接口，再把答案返回。”

这句话不能说错，但它只描述了最薄的一层表象。真正能上线、能维护、能扩展的聊天系统，至少还要回答下面这些问题：

1. 用户这次提问属于哪个会话？
2. 为什么不同用户的历史消息不能串掉？
3. 新会话和老会话的处理流程为什么不同？
4. 流式输出到底怎么落地？
5. 历史消息应该从哪里读，内存、数据库还是缓存？
6. 为什么聊天不仅是“调一次模型”，还会牵扯消息编排、RAG、MCP、异步落库和启动恢复？

所以这一篇的目标，不是“做一个能返回答案的聊天按钮”，而是从 0 到 1 搭建一个真正有工程边界的会话与聊天模块。

这一轮在项目里对应的核心改造包括：

- 把聊天主链路抽到 `internal/modules/chat/service.go`
- 让 `service/session/session.go` 退化成兼容包装层
- 会话列表改为从数据库读取，而不是只看内存
- 历史消息改为按 `IsUser` 字段还原，不再靠奇偶下标猜测角色
- 给 `session` 持久化 `model_type`
- 增加会话归属校验，避免跨用户访问别人的会话
- 给会话增加“最近活跃时间”更新能力，让列表排序更符合真实使用习惯
- 启动恢复阶段按会话维度恢复 AIHelper，而不是无脑用默认模型
- 修复 `AIHelper.AddMessage` 并发写消息缺少加锁的问题

这一步的本质，是把“聊天功能”提升为“聊天模块”。

## 二、先看整体流程：一条消息是怎么穿过整条链路的

### 1. 新会话的普通请求流程

```text
用户输入问题并选择模型
-> 前端调用 POST /AI/chat/send-new-session
-> Controller 绑定请求参数并从鉴权上下文取当前用户
-> Chat 模块创建 session 记录，保存 title 和 model_type
-> Chat 模块创建 user+session 维度的 AIHelper
-> 用户消息先写入内存，再异步投递到 RabbitMQ
-> AIModel 生成回复
-> 助手消息再写入内存，并异步投递到 RabbitMQ
-> touchSession 更新 session.updated_at
-> 返回 sessionId 和本次回答
```

这一条链路里最重要的动作，不是“调模型”，而是先把会话边界建起来。因为只有先知道这条消息属于哪一个 session，后面你才能谈历史上下文、历史回放、模型切换、知识库隔离和多轮对话。

### 2. 已有会话的流式请求流程

```text
用户在已有会话中继续提问
-> 前端调用 POST /AI/chat/send-stream
-> Controller 绑定参数并透传 request context
-> Chat 模块先校验 session 是否存在，以及是否属于当前用户
-> 若内存里没有该会话的 AIHelper，则按 session + DB 历史消息恢复一个热会话
-> 进入模型流式输出
-> 每收到一个 chunk，就按 SSE 格式写 data: ... 并 Flush
-> 前端通过 fetch + ReadableStream.getReader() 持续读取
-> 收到 [DONE] 后结束本次流式渲染
-> session.updated_at 刷新为最新活跃时间
```

这里你会看到一个很典型的工程现象：

**聊天不是一次请求返回一个字符串，而是一条贯穿前端渲染、后端流式传输、模型流、消息持久化、会话恢复的完整管线。**

### 3. 冷会话和热会话为什么处理方式不一样

项目里当前把聊天历史分成两类状态：

- 热状态：当前进程内已经活跃的 `AIHelper`
- 冷状态：已经落到数据库中的 `message` 记录

所以读取历史消息时，应该优先用热状态保证“当前会话最新”，热状态不存在时再从数据库回放。这正是这次重构里 `GetChatHistory` 的处理思路。

## 三、项目里该怎么实现这部分能力

### 1. 为什么会话必须单独建表

如果只保存 `message` 表而没有 `session` 表，项目会遇到几个问题：

- 很难高效列出“当前用户有哪些会话”
- 很难维护会话标题
- 很难保存会话级配置，比如当前会话使用哪个模型
- 很难做最近活跃排序
- 后面接入会话摘要、置顶、归档、删除等能力时会越来越难改

因此会话应该是第一类实体，而不是消息的副产品。

这次重构后，`model.Session` 的语义更完整了：

```go
type Session struct {
    ID        string
    UserName  string
    Title     string
    ModelType string
    CreatedAt time.Time
    UpdatedAt time.Time
}
```

这几个字段分别承担不同职责：

- `ID`：会话唯一标识
- `UserName`：会话归属用户
- `Title`：列表展示标题
- `ModelType`：该会话绑定的模型类型
- `UpdatedAt`：最近活跃时间，用于列表排序

### 2. 为什么把聊天主链路抽到 `internal/modules/chat`

这次没有继续把会话逻辑散落在 `controller/service/dao/common` 里处理，而是先建立了聊天模块边界：

- `internal/modules/chat/service.go` 负责聊天主流程
- `service/session/session.go` 只保留兼容包装
- `controller/session/session.go` 负责 HTTP 入参和响应
- `dao/session/session.go`、`dao/message/message.go` 负责存储读写
- `common/aihelper` 负责热会话状态与模型调用
- `internal/bootstrap/bootstrap.go` 负责启动恢复

这样做的好处是，后面无论你继续加：

- 会话摘要
- 历史压缩
- token 统计
- 会话分享
- 多模型路由
- 会话级 RAG 策略

都可以沿着 `chat` 这个模块边界继续往里长，而不必再回到“到处找聊天代码”的状态。

### 3. 为什么要把 `model_type` 持久化到 `session`

这是这次改造里非常关键的一点。

原来的问题是：

- 创建会话时选了哪个模型，没有稳定持久化
- 服务重启后恢复历史消息，只能用默认模型去恢复 AIHelper
- 同一个 session 后续继续聊天时，如果前端传了另一个 `modelType`，会话到底该用哪个模型并不清晰

现在把 `model_type` 放进 `session` 表以后，规则就明确了：

1. 新建会话时把模型类型存下来。
2. 旧会话继续聊天时，以会话自己的 `model_type` 为准。
3. 启动恢复热会话时，也按照 `session.model_type` 创建对应的 AIHelper。

这其实是在做一件很工程的事情：

**把“会话的运行配置”从一次请求里的临时参数，提升成持久化的会话元数据。**

### 4. 为什么会话列表不能只从内存拿

旧逻辑里，会话列表是从 `AIHelperManager` 的内存 map 里取的，也就是：

- 进程里有，就能看到
- 进程里没有，就看不到

这会带来几个现实问题：

- 服务刚重启但恢复不完整时，列表可能缺会话
- 某个会话之前落库了，但当前进程没有热状态，就会“消失”
- 会话标题也拿不到，只能临时用 sessionID 充数

所以这次把会话列表改成数据库查询，并按 `updated_at desc` 排序。数据库才是这里的事实来源，内存只是运行时加速层。

### 5. 历史消息为什么不能再用奇偶下标判断角色

这是原链路里一个很容易被忽视、但实际风险很高的问题。

如果你这么写：

```go
isUser := i%2 == 0
```

表面上看，在“严格一问一答”的理想场景里好像没问题。但一旦出现下面任意一种情况，就会错：

- 系统消息插入
- 工具调用消息插入
- 历史恢复顺序变化
- 某一轮只有用户消息或只有助手消息
- 未来加入多角色消息

所以正确做法应该是直接使用 `message.IsUser` 字段：

```go
history = append(history, model.History{
    IsUser:  msg.IsUser,
    Content: msg.Content,
})
```

这是一个非常典型的工程意识：

**有显式语义字段时，不要用位置关系去猜。**

### 6. 为什么聊天链路需要“内存 + MQ + 数据库”三层

GopherAI 现在的消息处理不是直接同步 `INSERT message`，而是：

1. 先写入 AIHelper 内存，保证当前上下文立刻可用。
2. 再通过 RabbitMQ 异步投递，后台消费者负责真正落库。
3. 需要冷启动恢复时，再从数据库把历史消息回放到 AIHelper。

这样分层的原因分别是：

- 内存：适合热状态，适合当前会话上下文拼装，适合立刻流式返回
- RabbitMQ：削峰解耦，避免主请求链路因为写库阻塞而变慢
- MySQL：长期持久化，适合历史查询、恢复和统计

所以这里不是为了“技术堆料”，而是让每一层只做自己最擅长的事。

### 7. 流式输出为什么用 SSE

这次流式聊天依然采用 SSE 思路输出，后端核心动作包括：

- 设置 `Content-Type: text/event-stream`
- 设置 `Cache-Control: no-cache`
- 禁止代理缓冲 `X-Accel-Buffering: no`
- 检查 `http.Flusher`
- 每写一段就 `Flush()`
- 最后发一个 `[DONE]`

我这次还额外补了一层 `writeSSEData()`，把多行内容拆开逐行写成 `data:`，避免 chunk 里本身带换行时破坏 SSE 协议格式。

```go
func writeSSEData(writer http.ResponseWriter, flusher http.Flusher, data string) error {
    normalized := strings.ReplaceAll(data, "\r\n", "\n")
    for _, line := range strings.Split(normalized, "\n") {
        if _, err := writer.Write([]byte("data: " + line + "\n")); err != nil {
            return err
        }
    }
    _, err := writer.Write([]byte("\n"))
    if err != nil {
        return err
    }
    flusher.Flush()
    return nil
}
```

### 8. 前端为什么不用 `EventSource`，而用 `fetch + ReadableStream`

这是非常适合面试追问的点。

SSE 在浏览器侧最经典的 API 是 `EventSource`，但当前项目的流式聊天仍然选择了：

- `fetch`
- `response.body.getReader()`
- 手动解析 `data:` 行

原因并不复杂：

1. 当前聊天接口是 `POST`，而 `EventSource` 更偏向简单 GET 长连接。
2. 这里需要传 JSON 请求体，比如 `question/sessionId/modelType`。
3. 通常还希望带鉴权头，`fetch` 的控制力更强。
4. 前端可以顺手处理首个 `sessionId` 事件、`[DONE]` 事件和错误事件。

所以不是“不会用 EventSource”，而是结合当前接口形态，`fetch + ReadableStream` 更灵活。

### 9. 为什么要把 `request context` 一路往下传

这次我把聊天主流程改成了显式接收 `context.Context`。

也就是说，Controller 不再只把参数传下去，而是把：

- `c.Request.Context()`

一起传到了聊天模块里。

这样做的意义是：

- 用户取消请求时，下游模型调用可以感知取消
- 流式连接断开时，后续工作更容易停止
- 后面接链路追踪、超时控制时更自然
- 更符合 Go 服务里“请求级上下文向下游传播”的最佳实践

这也是为什么真正的 Go 工程里，`context` 不是“可有可无的额外参数”，而是请求生命周期的一部分。

## 四、为什么这次要这样重构，而不是继续沿着旧逻辑补丁式修修补补

### 1. 因为聊天已经是业务模块，不是一个接口

当项目开始有下面这些能力时，聊天就已经不是单个接口了：

- 新建会话
- 继续会话
- 流式输出
- 历史回放
- 模型切换
- 文件上传后接 RAG
- 工具调用后接 MCP
- 启动恢复热状态

这意味着聊天必须有自己的模块边界，而不是继续被动地散落在很多目录里。

### 2. 因为“会话边界”比“模型调用”更先决定系统正确性

很多同学容易把注意力全放在模型上，但对工程系统来说，先问清楚这件事更重要：

**这条消息到底属于谁、属于哪个 session、应该使用哪段历史、应该落到哪份数据上。**

只有这些边界明确了，模型回复才有意义。

### 3. 因为热状态和冷状态必须分离

如果只靠数据库，每次拼上下文都很重；如果只靠内存，服务一重启历史就没了。

所以合理做法不是二选一，而是：

- 热会话在内存里
- 长期历史在数据库里
- 异步链路用 MQ 解耦

这其实就是典型的“运行时状态”和“持久化状态”分层。

## 五、实现中必须特别注意的坑

### 1. 会话列表不能只靠内存

内存只是运行时状态，不是事实来源。事实来源应该是数据库。

### 2. 历史消息角色不能靠奇偶位猜

必须依赖显式字段，比如 `IsUser`。

### 3. 必须做会话归属校验

只根据 `sessionId` 查到会话还不够，还必须校验这个会话是否属于当前登录用户。否则就会出现典型的水平越权。

### 4. `model_type` 必须会话级持久化

否则重启恢复、续聊、模型一致性都会出问题。

### 5. 异步 MQ 落库一定会带来最终一致性

也就是说，热会话里已经有最新消息了，但数据库可能还没来得及写进去。所以聊天历史查询不能只依赖数据库，也不能只依赖内存，要按冷热分层来读。

### 6. 流式输出一定要考虑协议细节

包括：

- 响应头是否正确
- 是否及时 Flush
- chunk 里带换行怎么办
- 客户端断连怎么办
- 结束标记怎么发

### 7. 并发写消息必须加锁

`AIHelper.AddMessage` 如果没有锁，流式响应和普通响应同时写同一个会话时，会埋下数据竞争风险。

### 8. 会话排序要用最近活跃时间，而不是只看创建时间

所以这次补了 `TouchSession`。对于聊天产品，用户真正关心的是“最近聊过的会话排在前面”，而不是“最早创建的会话排在前面”。

## 六、围绕本篇内容，需要进一步掌握哪些技术基础

### 1. 会话建模

至少要分清楚两类实体：

- `session`：会话级元数据
- `message`：消息级记录

不要把两者混在一起。

### 2. Go 的 `context.Context`

你要知道它至少解决三件事：

- 超时
- 取消
- 请求级上下文传播

### 3. SSE 与 WebSocket 的区别

SSE 更适合：

- 服务端单向推送文本流
- 流式回复
- 实现成本较低的聊天输出

WebSocket 更适合：

- 双向实时通信
- 高频交互
- 更复杂的实时协作场景

### 4. `ReadableStream` 和增量渲染

前端流式体验的本质不是“后端支持流”，而是浏览器能不能一边收、一边解析、一边渲染。

### 5. MQ 与最终一致性

异步化能提升主链路响应速度，但一定要明白它换来的是最终一致性，而不是强一致。

### 6. 聊天为什么不只是一次 LLM 调用

GopherAI 当前模型工厂里已经有多种实现：

- OpenAI 对话
- Ollama 本地模型
- RAG 检索增强
- MCP 工具调用

这意味着“聊天”这件事在系统里其实已经可能变成：

- 拼历史消息
- 做检索增强
- 判断是否需要调用工具
- 再进入二次模型生成

所以以后如果面试官追问你“聊天为什么不只是调用一次模型接口”，你要能把这条链路讲出来。

## 八、提问 + 标准答案

下面这些问题，既适合你做项目复盘，也很接近国内大中厂常见追问。

### 问题 1：为什么聊天系统里要单独建 `session` 表？

**标准答案：**

因为 `session` 和 `message` 不是同一层数据。`message` 负责记录具体消息内容，`session` 负责承载会话级元数据，例如会话归属、标题、模型类型、最近活跃时间等。如果没有 `session` 表，会话列表、标题管理、模型绑定、最近活跃排序、会话级配置都会很难维护。

### 问题 2：为什么会话列表不能只从内存里拿？

**标准答案：**

因为内存只是运行时热状态，不是最终事实来源。服务重启、热恢复不完整、某些会话还没被加载进内存时，只看内存就会漏数据。会话列表应该以数据库为准，内存只负责加速当前活跃会话。

### 问题 3：为什么历史消息不能用奇偶下标判断用户和助手？

**标准答案：**

因为奇偶位只适用于非常理想的一问一答序列。一旦引入系统消息、工具消息、消息缺失、恢复顺序变化或者多角色消息，奇偶位就会失真。正确做法是使用显式字段，比如 `IsUser`，因为它表达的是语义，而不是位置猜测。

### 问题 4：为什么必须做会话归属校验？

**标准答案：**

因为只校验 `sessionId` 是否存在，不校验其归属用户，就可能导致用户通过猜测或拿到别人的 `sessionId` 访问他人的会话，这属于典型的水平越权。正确做法是同时校验“会话存在”与“会话归属于当前登录用户”。

### 问题 5：为什么 `model_type` 适合放在 `session` 上，而不是每次请求临时传？

**标准答案：**

因为模型选择本质上是会话级配置，而不是单次请求的随机参数。如果只靠每次请求临时传，会导致重启恢复、续聊、多轮一致性都不稳定。把 `model_type` 存到 `session` 后，历史恢复、继续会话、流式续聊都能保持一致语义。

### 问题 6：为什么聊天链路里既要内存，也要数据库？

**标准答案：**

因为两者职责不同。内存适合热状态，读取快，适合当前上下文拼装和流式场景；数据库适合长期持久化、恢复和统计。只用数据库会让当前会话变重，只用内存会让服务重启后历史丢失。合理方案是冷热分层。

### 问题 7：为什么还要引入 RabbitMQ，而不是同步写数据库？

**标准答案：**

因为消息落库属于可以异步化的操作。把它放到 MQ 后面，可以降低主请求链路延迟，避免模型回复刚出来时还要同步等待数据库写入完成。代价是系统从强一致转成最终一致，所以查询链路也要配套设计冷热读策略。

### 问题 8：异步落库最大的代价是什么？

**标准答案：**

最大的代价是最终一致性，也就是内存里已经有最新消息了，但数据库可能暂时还没有。除此之外，还要额外考虑消费者失败重试、消息幂等、积压监控和死信处理等问题。

### 问题 9：为什么这里的流式聊天适合用 SSE？

**标准答案：**

因为当前场景本质上是服务端单向持续推送文本块给前端，SSE 足够简单，浏览器支持也好，服务端基于 HTTP 就能实现。相比之下，WebSocket 更适合真正需要双向高频实时通信的场景。

### 问题 10：那为什么前端没用 `EventSource`，而是用了 `fetch + ReadableStream`？

**标准答案：**

因为当前接口是 `POST` 请求，而且需要携带 JSON 请求体和鉴权头。`fetch` 在这类场景下更灵活，前端还能手动处理首个 `sessionId` 事件、`[DONE]` 标记和错误分支，所以并不是 SSE 只能配 `EventSource`。

### 问题 11：`http.Flusher` 的作用是什么？

**标准答案：**

它的作用是让服务端把当前缓冲区里的数据立即刷给客户端，而不是等整个响应完成后再一起发送。流式输出如果不 `Flush()`，前端就感受不到“边生成边显示”的效果。

### 问题 12：为什么服务端写 SSE 时要按行写 `data:`？

**标准答案：**

因为 SSE 协议本身就是按文本行组织的。如果一个 chunk 里本身带换行，直接原样写出去可能破坏事件边界。更稳妥的做法是把文本拆成多行，每行都加 `data:` 前缀，最后再补一个空行表示一个事件结束。

### 问题 13：为什么这次把 `context.Context` 显式传到了聊天模块？

**标准答案：**

因为请求级上下文应该贯穿 Controller、Service、模型调用和流式输出。这样用户取消请求、连接中断或超时时，下游逻辑可以感知并尽早停止，避免无意义的模型调用和资源浪费。

### 问题 14：为什么 `session.updated_at` 要随着发消息更新？

**标准答案：**

因为对聊天产品来说，会话列表通常按最近活跃排序更符合用户认知。如果只记录创建时间，不更新活跃时间，那么用户刚聊过的会话可能还排在很后面，体验会很差。

### 问题 15：服务重启以后，为什么还要做 `WarmAIHelpers`？

**标准答案：**

因为当前聊天上下文拼装依赖 AIHelper 的热状态。如果服务重启后完全不恢复热状态，那么老会话第一次继续聊天时，要么拿不到上下文，要么每次都临时从数据库重新组装。预热恢复能让热点会话更快回到可用状态。

### 问题 16：为什么 AIHelper 要按“用户 + 会话”维度管理？

**标准答案：**

因为会话上下文天然属于某个用户的某个 session。按这个维度管理，可以把不同用户、不同会话的消息隔离开，避免上下文串线，也便于后续回收、统计和限流。

### 问题 17：为什么说“聊天不只是一次模型调用”？

**标准答案：**

因为真正的 AI 应用里，聊天往往还包含历史上下文拼接、RAG 检索增强、工具调用判断、工具结果注入、二次生成等步骤。像 GopherAI 当前就已经有 OpenAI、Ollama、RAG、MCP 等多种模型分支，所以聊天链路本质上是编排问题，不只是 HTTP 转发。

### 问题 18：如果消息越来越多，上下文窗口怎么办？

**标准答案：**

不能无限制把所有历史消息都塞进模型。常见做法包括：窗口裁剪、会话摘要、重要消息提取、按主题归档、RAG 检索补充历史等。也就是说，会话历史的“持久化完整保留”和“模型输入上下文适度裁剪”是两件事。

### 问题 19：如果面试官问你“MQ 异步落库是不是就一定更好”，你怎么回答？

**标准答案：**

不会这么绝对。MQ 异步化能降低主链路延迟、增强解耦，但也会带来最终一致性、消费失败、幂等、监控和运维复杂度。是否值得引入，要看系统的吞吐、时延要求和复杂度预算。
