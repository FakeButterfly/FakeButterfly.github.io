## 一、为什么先做认证与启动层

如果要从零开始搭一个 AI 应用，很多人会先去接模型、先写聊天窗口、先跑一个能对话的 Demo。

但只要项目开始往“多人使用、持续演进”的方向走，两个问题就会立刻冒出来：

1. 谁在访问系统？
2. 系统是按什么顺序跑起来的？

这两个问题对应的，就是认证层和启动层。

在 `GopherAI` 里，后续会涉及聊天、文件上传、RAG、图像识别、语音合成、MCP 等能力。如果没有统一的认证层，就很难做用户隔离、会话归属、知识库归属；如果没有清晰的启动层，后面加 MySQL、Redis、RabbitMQ、模型客户端时，`main.go` 很快就会失控。

所以第一篇不先写“炫”的功能，而是先把骨架搭稳。

## 二、这一部分在项目里的完整流程是什么

### 1. 登录注册流程

以登录为例，一次完整请求通常会经过下面几步：

1. 前端提交账号和密码。
2. 控制器接收请求并做参数绑定。
3. 用户服务校验账号信息。
4. 校验通过后，服务端签发 JWT。
5. 前端保存 JWT，并在后续请求里放到 `Authorization` 请求头中。

这个流程的核心结果，不是“登录成功”四个字，而是服务端和客户端建立起一个可验证的身份凭证。

### 2. 鉴权请求流程

后续访问需要登录的接口时，请求链路会变成：

1. 请求进入 Gin 路由。
2. JWT 中间件从 Header 中提取 Bearer Token。
3. 中间件校验 token 是否合法、是否过期、签名算法是否正确。
4. 校验通过后，把当前用户身份写入请求上下文。
5. 控制器从上下文拿到当前用户，再进入业务逻辑。

这样做的好处是，控制器本身不需要反复关心 token 怎么解析，它只需要关心“当前用户是谁”。

### 3. 应用启动流程

应用启动时的链路，则更适合按下面顺序组织：

1. 读取配置。
2. 初始化 MySQL。
3. 预热或恢复 AI 助手相关内存状态。
4. 初始化 Redis。
5. 初始化 RabbitMQ。
6. 注册路由并启动 HTTP 服务。

把这些动作直接堆在 `main.go` 里虽然也能跑，但不利于维护。更好的方式，是把这条链路放进独立的 `bootstrap` 模块里统一管理。

## 三、代码应该如何实现

### 1. 认证模块拆分

一个最小但足够清晰的认证模块，建议至少拆成三层：

#### `token.go`

负责：

- 生成 JWT
- 解析 JWT
- 校验签名算法
- 处理过期时间与密钥配置

这一层只关心 token 本身，不关心 Gin，不关心业务。

#### `middleware.go`

负责：

- 从请求头中提取 Bearer Token
- 调用 `token.go` 校验 token
- 鉴权成功后把用户信息写入上下文
- 鉴权失败时直接拦截请求

这一层是 HTTP 入口和认证逻辑之间的桥梁。

#### `context.go`

负责：

- 统一读取当前登录用户
- 避免业务代码到处写 `c.GetString("userName")`

很多项目后期会越来越乱，一个常见原因就是上下文字段名散落在各处。抽一层 helper，本质上是在减少业务对框架细节的耦合。

### 2. 启动模块拆分

`main.go` 最好只保留入口职责：

- 调用 bootstrap
- 注册路由
- 启动服务

真正的初始化动作，放进 `bootstrap.go` 中，例如：

- `InitMySQL()`
- `InitRedis()`
- `InitRabbitMQ()`
- `RestoreAgentMemory()`

这样一来，应用启动就从“杂糅在入口文件里的过程代码”，变成了“可维护的启动流程”。

## 四、为什么这样实现

### 1. 为什么认证逻辑不能散落在控制器里

如果每个控制器都自己去解析 token，会有三个明显问题：

- 重复代码很多
- 容易出现不同接口鉴权标准不一致
- 后面一旦要改 token 规则，改动面会非常大

中间件的价值就在于把这部分逻辑前置并统一。

### 2. 为什么不建议到处直接读 Gin 上下文字符串

`c.GetString("userName")` 这种写法短期看很方便，但长期看有两个问题：

- 魔法字符串没有约束，写错了运行时才暴露
- 框架上下文和业务逻辑耦合过深

抽成 `GetCurrentUser()` 这类 helper 后，控制器只依赖一个稳定接口，而不是到处复制字符串键名。

### 3. 为什么 `main.go` 不能什么都做

`main.go` 是程序入口，但不应该是所有初始化逻辑的垃圾桶。

当一个项目开始接入数据库、缓存、消息队列、对象存储、模型客户端后，如果入口文件不收敛，后面无论排查问题还是做环境切换都会越来越困难。

## 五、实现中应该注意哪些坑

### 1. JWT 解析时的空指针风险

这是一个很经典的坑。

很多人第一次写 JWT 校验时，会直接判断 `token.Valid`。但如果解析失败，`token` 本身可能就是 `nil`，这时候继续访问它会直接 panic。

更稳妥的顺序应该是：

1. 判断 token 字符串是否为空。
2. 调用解析函数。
3. 先判断 `err`。
4. 再判断 `token == nil`。
5. 最后再判断 `token.Valid`。

### 2. 不要在日志里打印完整 token

JWT 本质上就是登录态凭证。如果把完整 token 打进日志，一旦日志泄漏，攻击者就可能直接复用这个凭证。

日志里应该记录：

- 鉴权失败原因
- 请求路径
- 用户名或用户 ID
- 必要的调试上下文

而不是直接打印原始 token。

### 3. JWT 有签名，不代表传输就安全

JWT 的签名保证的是“内容没有被篡改”，但不保证“传输过程中不会被偷走”。

所以它必须配合 HTTPS 使用。

### 4. 不要把密码哈希理解成“随便 MD5 一下”

在教学型项目里，很多人会先用 MD5 跑通流程。但真正上线项目里，密码存储更应该考虑：

- 加盐
- 慢哈希
- 专门的密码算法，例如 `bcrypt`、`scrypt`、`argon2`

否则即使业务流程是对的，安全性也远远不够。

## 六、这一部分涉及到的基础知识

### 1. 什么是 JWT

JWT 全称是 JSON Web Token，本质上是一种用于传递身份声明的 token 格式。

它通常由三部分组成：

- Header
- Payload
- Signature

其中：

- Header 说明签名算法等元信息
- Payload 存放用户标识、过期时间等声明
- Signature 用来防篡改

### 2. JWT 为什么适合前后端分离

前后端分离项目里，前端和后端天然是通过 API 协作的。JWT 放在请求头里传递身份，比较适合这种架构。

它的优势主要在于：

- 服务端无状态
- 方便多实例扩容
- 适合通过网关和服务间透传身份

### 3. JWT 和 Session 的区别

JWT 并不是天然比 Session 高级，它只是更适合某些架构场景。

如果项目更需要：

- 服务端强控会话
- 立即失效登录态
- 方便踢人下线

那么 Session 反而可能更合适。

### 4. HTTPS 是如何保护 JWT 的

HTTPS 本质上是 `HTTP + TLS`。

TLS 大致做三件事：

1. 服务端通过证书证明自己是谁。
2. 客户端和服务端协商会话密钥。
3. 后续业务数据用对称加密传输。

所以可以记住一句话：

- JWT 解决身份声明可信
- HTTPS 解决传输过程安全

这两者不是替代关系，而是配合关系。

## 七、提问 + 标准答案

### 问题 1：为什么一个 AI 应用项目要先做认证层？

**标准答案：**

因为只要项目涉及会话历史、文件上传、知识库归属、用户级配置，就必须先区分“谁在访问系统”。认证层本质上是在建立用户边界，没有这个边界，后面的聊天、RAG、上传等能力都很难真正落地。

### 问题 2：为什么这里选择 JWT，而不是 Session？

**标准答案：**

因为当前项目是前后端分离架构，后续还有可能扩展为多模块甚至多服务，JWT 更适合通过 Header 做身份透传，也更利于横向扩容。但这不代表 JWT 一定优于 Session，Session 在服务端强控会话、踢人下线、立即失效方面反而更方便。

### 问题 3：JWT 为什么一定要配合 HTTPS？

**标准答案：**

JWT 的签名只能防止内容被篡改，不能防止 token 在传输过程中被截获。HTTPS 通过 TLS 保护传输链路，避免 token 被中间人窃听，所以 JWT 和 HTTPS 需要配合使用。

### 问题 4：为什么要把鉴权逻辑放进中间件？

**标准答案：**

因为鉴权是横切逻辑，多个接口都要复用。如果散落在控制器中，会造成重复代码和行为不一致。放进中间件后，可以统一处理 token 提取、校验、拦截和上下文注入。

### 问题 5：为什么要把启动流程从 `main.go` 中拆出来？

**标准答案：**

因为 `main.go` 的职责应该是应用入口，而不是所有初始化逻辑的堆放点。把 MySQL、Redis、RabbitMQ、模型预热等过程抽到 `bootstrap` 后，启动顺序更清晰，也更利于后续扩展和排障。

## 八、这一篇学完之后，你应该能掌握什么

学完这一篇，至少应该能说清楚下面这些问题：

- 为什么认证层和启动层适合作为 AI 项目的第一阶段骨架
- JWT 的组成和适用场景是什么
- Gin 中间件在鉴权链路里做了什么
- 为什么 `main.go` 不应该堆满初始化逻辑
- JWT、HTTPS、密码存储之间分别解决什么问题
