## 一、为什么第一篇先写认证与启动层

如果是第一次做 AI 应用，很多人会本能地先去接模型、先做聊天窗口、先让页面跑出一句回答。

但真正把项目往“长期维护、多人协作、持续扩展”的方向推进时，最先暴露问题的往往不是模型，而是下面两件事：

1. 谁在访问系统，也就是身份边界。
2. 系统按什么顺序启动，也就是运行边界。

这两件事分别对应认证层和启动层。

对 `GopherAI` 来说，后面会持续增加：

- 用户登录与注册
- AI 聊天与会话管理
- 文件上传
- RAG 检索增强
- 图像识别
- 语音合成
- MCP 工具调用

只要项目开始涉及“用户私有数据”和“多个基础组件协作”，认证层和启动层就必须先稳定下来。否则后面的聊天、知识库、文件隔离、工具调用都会越来越乱。

所以这一篇虽然不花哨，但它改的是骨架。

## 二、这一部分在项目中的完整流程

### 1. 登录与注册流程

从业务角度看，登录链路并不复杂，但每一步都很关键：

1. 前端提交账号、密码、验证码等字段。
2. 控制器接收请求并做参数绑定与基础校验。
3. 用户服务层查询用户信息，并校验密码或验证码。
4. 校验通过后，服务端生成 JWT。
5. 前端保存 JWT，并在后续请求中通过 `Authorization: Bearer <token>` 携带。

这个流程的核心结果并不是“登录成功”，而是建立了一个可被服务端验证的身份凭证。

### 2. 鉴权请求流程

登录完成之后，访问受保护接口通常会经过下面这条链路：

1. 请求进入 Gin 路由。
2. JWT 中间件从 `Authorization` 头中提取 Bearer Token。
3. 中间件校验 token 是否为空、是否过期、签名是否合法、算法是否符合预期。
4. 校验通过后，把当前用户信息写入请求上下文。
5. 后续控制器和服务层读取当前用户，执行业务逻辑。

也就是说，控制器真正关心的应该是“当前用户是谁”，而不是“token 该怎么解析”。

### 3. 应用启动流程

启动链路适合统一放进 `bootstrap`，而不是零散堆在 `main.go`：

1. 读取配置并初始化配置对象。
2. 初始化 MySQL。
3. 恢复或预热 AI 助手相关内存状态。
4. 初始化 Redis。
5. 初始化 RabbitMQ。
6. 注册路由并启动 HTTP 服务。

如果后面再加入日志、指标、链路追踪、配置中心、对象存储，依然可以沿着这条链路扩展，而不会让入口文件失控。

## 三、代码应该如何实现

### 1. 认证模块应该拆成哪几层

这一轮我把认证模块抽成了三个最核心的职责：

#### `token.go`

负责：

- 生成 JWT
- 解析 JWT
- 校验签名算法
- 处理过期时间、签发者、密钥等配置

它只处理 token 本身，不依赖具体业务。

#### `middleware.go`

负责：

- 从请求头提取 Bearer Token
- 调用 token 解析逻辑
- 鉴权失败时直接拦截请求
- 鉴权成功后，把用户信息放进上下文

这一层是 HTTP 请求入口和认证能力之间的桥梁。

#### `context.go`

负责：

- 统一读取当前登录用户
- 封装上下文中的用户键名
- 避免业务层到处写 `c.GetString("userName")`

这一步非常重要，因为它本质上是在避免“魔法字符串”扩散。

### 2. 启动模块应该怎么拆

`main.go` 最好尽量只保留入口职责：

- 加载配置
- 调用 bootstrap
- 创建路由
- 启动服务

而真正的初始化细节，例如：

- MySQL 初始化
- Redis 初始化
- RabbitMQ 初始化
- AI 助手状态恢复

都集中放进 `bootstrap.go`，这样有几个好处：

- 启动顺序清晰
- 依赖关系更明确
- 更容易做故障排查
- 更容易扩展健康检查和优雅关闭

### 3. 为什么现在就要建立模块意识

很多项目一开始按 `controller/service/dao/router` 分目录，并不是错。

问题在于，当业务越来越多时，同一个功能会散落到很多目录里。一个认证模块，可能同时横跨控制器、中间件、工具包、路由、服务层。到了后期，改一个认证逻辑要来回跳很多文件。

因此，即使还没有“一次性重构完全”，也应该从最先稳定的核心能力开始建立业务模块意识。认证层和启动层就是最合适的第一刀。

## 四、为什么要这样实现

### 1. 为什么鉴权逻辑要放中间件

因为鉴权是典型的横切逻辑。

如果每个控制器都手动解析 token，会有几个明显问题：

- 重复代码很多
- 每个接口的处理方式可能不一致
- 后面一旦修改鉴权规则，改动面会非常大

中间件的本质就是把这种“很多接口都要做、但又不属于某个具体业务”的逻辑前置并统一。

### 2. 为什么用户信息要放上下文 helper，而不是直接散落在 Gin context 中

短期看，直接写 `c.GetString("userName")` 很省事。

但长期看有两个问题：

- 键名是魔法字符串，写错了编译器不会帮你发现。
- 业务逻辑会强依赖 Gin 的上下文实现细节。

抽一层 `GetCurrentUser()` 的价值，在于把“框架上下文”和“业务语义”分开。

### 3. 为什么 `main.go` 不能成为大杂烩

`main.go` 是程序入口，但它不应该承担“所有初始化细节的容器”这个职责。

一个工程项目里，入口文件最适合做的是编排，不适合做复杂实现。入口层越干净，后面排障、扩展、环境切换和测试越轻松。

## 五、实现中容易踩的坑

### 1. JWT 解析时的空指针风险

很多人第一次写 JWT 校验会这么处理：

- 调用解析函数
- 直接判断 `token.Valid`

问题在于，解析失败时 `token` 可能本身就是 `nil`，这时候再访问它就会 panic。

更稳妥的顺序应该是：

1. 先判断 token 字符串是否为空。
2. 再调用解析函数。
3. 先处理 `err`。
4. 再判断 `token == nil`。
5. 最后才判断 `token.Valid`。

### 2. 必须显式校验签名算法

JWT 的安全实现里，一个常见高频点就是“算法校验”。

不要假设库已经帮你把所有算法相关问题都完全兜底了。服务端应该明确限制允许的算法，例如这里只接受 `HS256`，而不是“只要能解出来就算合法”。

这也是 [RFC 8725](https://www.rfc-editor.org/rfc/rfc8725) 特别强调的安全实践之一。

### 3. JWT Payload 不是加密内容

很多初学者会误以为：JWT 看起来是一串“看不懂”的字符串，所以里面的数据是加密的。

其实不是。

绝大多数常见 JWT 只是 Base64URL 编码加签名保护，重点在于：

- 防篡改
- 可验证来源

它并不等于“对 Payload 做了保密加密”。所以：

- 不要把敏感隐私信息直接塞进 Payload
- 不要把密码、银行卡、完整权限细节等敏感信息放进 JWT

### 4. Bearer Token 不要随便放到 URL 上

项目里最常见、也最稳妥的方式，是把 token 放进 `Authorization` 请求头。

如果把 token 放到 URL query 参数里，会有明显风险：

- 浏览器历史记录泄漏
- 网关与代理日志泄漏
- Referer 传播
- 缓存系统记录 URL

这也是为什么 Bearer Token 在 HTTP 场景中，标准写法优先是 `Authorization: Bearer <token>`，[RFC 6750](https://www.rfc-editor.org/rfc/rfc6750) 对此有明确规定。

### 5. 不能在日志里打印完整 token

JWT 本身就是凭证。完整打印到日志里，一旦日志系统被查看、转储或泄漏，攻击者可能直接复用这个 token。

日志更适合记录：

- 失败原因
- 请求路径
- 用户 ID 或用户名
- 请求来源与时间

而不是原始 token 全量内容。

### 6. HTTPS 不能省

JWT 的签名只能说明“内容没被篡改”，不能说明“传输过程中不会被窃听”。

如果没有 HTTPS，token 依然有可能被中间人截获。Bearer Token 的特点就是“谁拿到，谁就能用”，所以 HTTPS 不是可选项，而是前提项。

### 7. 密码存储不能停留在 MD5

在教学型 Demo 里，很多人会先用 MD5 跑通注册登录流程。但实际工程里，MD5 这种快速哈希已经不适合密码存储。

更合理的做法是使用专门的密码哈希算法，例如：

- `bcrypt`
- `scrypt`
- `argon2`

并结合：

- 盐值
- 成本参数
- 后续升级策略

### 8. 启动顺序要 fail-fast

数据库、缓存、消息队列、配置中心这些基础组件，如果在启动阶段就失败了，很多服务根本不应该继续启动。

一个成熟一点的启动流程应该明确：

- 哪些组件失败必须中断启动
- 哪些组件失败可以降级
- 哪些组件需要在启动后持续健康检查

这就是所谓的 fail-fast 思维。

## 六、围绕本篇涉及到的技术，应该进一步理解什么

### 1. JWT、JWS、JWE 之间是什么关系

面试里很多人会把 JWT 直接理解成“加密 token”。

更准确地说：

- JWT 是一种承载声明的 token 格式
- JWS 关注签名
- JWE 关注加密

我们在很多项目里常用的 JWT，本质上通常是“带签名的声明”，不是“默认加密的密文”。

### 2. 为什么说 JWT 更适合前后端分离，但不等于一定优于 Session

JWT 的典型优势是：

- 服务端相对无状态
- 更适合 REST API 和网关透传
- 对多实例扩容更友好

但它也有明显边界：

- token 一旦签发，天然不如服务端 Session 好控
- 退出登录、强制下线、立即失效会更麻烦
- 一旦被盗，风险会持续到 token 过期或被服务端拉黑

所以成熟的回答应该是：JWT 不一定更高级，只是更适合当前架构。

### 3. 为什么很多项目用了 JWT，还要接 Redis

这是国内面试非常喜欢追问的一题。

JWT 本身可以无状态，但项目里仍然很可能接 Redis，因为 Redis 能补足 JWT 的一些“可控性”能力，例如：

- refresh token 存储
- token 黑名单
- 用户登录态版本号
- 验证码
- 短时限流
- 风险控制标记

所以“用了 JWT 就完全不需要 Redis”这个说法并不成立。

### 4. HTTPS 和 TLS 到底在做什么

可以先把概念分开：

- HTTPS 是“跑在 TLS 之上的 HTTP”
- TLS 是真正负责安全通信的协议

TLS 在握手阶段主要做三件事：

1. 服务端通过证书证明自己是谁。
2. 客户端与服务端协商会话密钥。
3. 后续业务数据使用对称加密传输。

为什么不是全程用非对称加密？因为非对称加密更适合做身份验证和密钥交换，而真正的数据传输量大，更适合交给对称加密。

### 5. HSTS 是什么，为什么很多人会忽略它

即使服务器已经配了 HTTP 到 HTTPS 的 301 跳转，第一次明文访问仍然可能被中间人劫持。

HSTS 的作用就是告诉浏览器：

- 以后这个站点只能用 HTTPS 访问
- 不允许用户轻易忽略证书错误

所以它是“HTTPS 已经启用之后”的进一步加固，而不是 HTTPS 的替代品。

### 6. 为什么 Go 项目里上下文传递也很关键

在 Go 服务里，一个请求往往会经过：

- HTTP Handler
- Service
- DAO
- Redis / MySQL / MQ 调用

如果没有 `context.Context`，请求超时、取消信号、trace 信息、请求级数据就很难一致地向下游传播。

这也是为什么后面项目继续往下做时，认证上下文、业务上下文、请求生命周期控制要一起考虑，而不是只盯着 Handler 一层。

## 八、结合项目与国内大中厂常见追问的面试题

下面这部分我按“提问 + 标准答案”的形式来写。你可以把它理解成项目复盘时的第一版题单。

### 问题 1：为什么在 GopherAI 里，第一阶段先做认证层和启动层，而不是先做聊天？

**标准答案：**

因为聊天、RAG、文件上传这些能力都依赖稳定的用户边界和基础组件边界。认证层解决“谁在访问系统”，启动层解决“系统如何稳定跑起来”。这两层不稳，后续功能再多也只是堆功能，难以工程化。

### 问题 2：JWT 的核心组成是什么？签名和加密有什么区别？

**标准答案：**

JWT 通常由 Header、Payload、Signature 三部分组成。签名的作用是防止内容被篡改，证明 token 来源可信；加密的作用是保护内容不被直接读取。常见项目里很多 JWT 只做了签名，不等于 Payload 天然保密。

### 问题 3：为什么项目里选择 JWT，而不是 Session？

**标准答案：**

因为当前项目是前后端分离架构，后续还可能扩展为多模块或多服务，JWT 更适合通过 Header 做身份透传，也更适合横向扩容。但这不代表 JWT 一定优于 Session，Session 在服务端强控登录态、踢人下线、立即失效方面更方便。

### 问题 4：为什么很多公司面试会继续追问“既然用了 JWT，为什么还需要 Redis”？

**标准答案：**

因为 JWT 解决的是“身份凭证可自包含”，不是“所有登录态管理问题都自动消失”。项目里仍然可能需要 Redis 来存 refresh token、维护黑名单、记录 token 版本、保存验证码和做限流，这些都属于 JWT 的外围控制能力。

### 问题 5：为什么 Bearer Token 通常放在 `Authorization` 头里，而不是 URL 参数中？

**标准答案：**

因为 URL 参数更容易被浏览器历史、代理日志、Referer、缓存系统记录下来，泄漏面更大。放在 `Authorization` 头中更符合规范，也更利于统一中间件处理。

### 问题 6：为什么 JWT 一定要配合 HTTPS？

**标准答案：**

因为 JWT 的签名只防篡改，不防窃听。Bearer Token 一旦被中间人截获，攻击者就能直接拿去使用。HTTPS 通过 TLS 保护传输链路，避免 token 在网络中裸奔。

### 问题 7：如果用户点击“退出登录”，但 JWT 还没过期，你怎么让它失效？

**标准答案：**

常见做法是短时 access token + refresh token 组合，再结合 Redis 黑名单、token version、设备级会话记录等方式做失效控制。也就是说，JWT 项目往往仍然会引入少量状态来换取更强的登录态可控性。

### 问题 8：为什么必须显式校验 JWT 的算法？

**标准答案：**

因为如果服务端不限制算法，只要“能解出来就算合法”，就可能引入算法混淆或错误接受风险。更安全的实现应该明确只接受允许的算法，例如 `HS256`，而不是完全信任 token 头里自带的算法声明。

### 问题 9：JWT 能不能防重放攻击？

**标准答案：**

不能单独完全防。JWT 的 `exp` 可以限制有效期，但如果 token 在有效期内被截获，攻击者依然能重放。真正缓解重放通常还需要 HTTPS、较短过期时间、敏感操作二次校验、`jti` 去重、nonce 或服务端状态辅助。

### 问题 10：为什么不建议把敏感信息直接放进 JWT Payload？

**标准答案：**

因为很多场景下 Payload 只是被编码和签名，不是被真正加密。任何拿到 token 的人都可能还原 Payload 内容，所以 Payload 应该尽量只放必要的身份声明，而不是敏感业务数据。

### 问题 11：为什么密码不能直接用 MD5？

**标准答案：**

因为 MD5 是快速哈希，太适合被暴力破解和撞库利用，不适合密码存储。密码存储应该使用专门的慢哈希算法，例如 `bcrypt`、`scrypt`、`argon2`，并结合成本参数和升级策略。

### 问题 12：如果面试官问“你用 Gin 中间件做了什么”，应该怎么回答？

**标准答案：**

可以回答：中间件负责统一提取 Bearer Token、做 JWT 校验、拦截非法请求，并在鉴权成功后把当前用户信息写入上下文。这样控制器只关心业务逻辑，不需要重复处理鉴权细节。

### 问题 13：为什么要把上下文用户信息封装成 helper，而不是直接在控制器里写字符串键？

**标准答案：**

因为直接写字符串键会让业务依赖魔法字符串和框架细节，长期维护风险很高。封装成 helper 之后，可以减少重复、提升可读性，也方便以后统一调整上下文字段和类型。

### 问题 14：如果数据库初始化失败，服务要不要继续启动？

**标准答案：**

取决于数据库是否是核心依赖。对于依赖数据库才能提供核心能力的服务，应该 fail-fast，直接终止启动并暴露错误，而不是带病运行。对于某些非核心依赖，可以视情况做降级。

### 问题 15：为什么启动流程适合做成 `bootstrap`，而不是继续堆在 `main.go`？

**标准答案：**

因为 `main.go` 更适合做应用编排，而不是细节实现。把 MySQL、Redis、MQ、模型预热等逻辑集中到 `bootstrap` 后，初始化顺序清晰、扩展性更好，也更方便后续增加健康检查和优雅关闭。

### 问题 16：Go 后端里为什么 `context.Context` 很重要？

**标准答案：**

因为一个请求会跨越 HTTP、Service、DAO、Redis、MySQL 等多层调用，`context` 用来统一传递取消信号、超时、trace 信息和请求级数据。如果没有它，请求超时后底层调用可能仍然继续执行，浪费资源。

### 问题 17：如果后面要把 GopherAI 从传统分层目录改成按业务模块组织，你会怎么推进？

**标准答案：**

我会先从高价值、边界比较清晰的模块开始抽离，比如认证、聊天、RAG、文件、工具调用等，而不是一次性大搬家。这样既能持续交付，也能逐步收敛目录结构和依赖方向。

### 问题 18：如果面试官问“JWT 和 Cookie 谁更安全”，怎么回答更成熟？

**标准答案：**

不能简单说谁绝对更安全。JWT 是 token 格式，Cookie 是浏览器存储和传输机制，两者不是同一维度。浏览器场景下，HttpOnly Cookie 对防 XSS 窃取有优势，但又要关注 CSRF；Header 传 token 更适合纯 API，但要关注前端存储方式。关键还是结合应用形态和安全边界来设计。

### 问题 19：为什么很多大厂喜欢继续追问 HSTS？

**标准答案：**

因为很多人只会说“我配了 HTTPS”，但忽略第一次明文访问仍有被劫持风险。HSTS 是浏览器侧强制 HTTPS 的机制，能让站点在后续访问中更安全，是 HTTPS 上线后的常见加固动作。

### 问题 20：如果让你再做一次这一部分，你会继续补哪些工程能力？

**标准答案：**

我会继续补：

- access token / refresh token 双 token 机制
- 更安全的密码哈希算法
- 统一配置校验
- 启动健康检查
- 优雅关闭
- 鉴权失败审计日志
- 用户登录设备管理与踢下线能力

这说明项目已经开始从“能跑”往“可控、可运维、可审计”演进。
