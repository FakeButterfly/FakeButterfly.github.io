# 第四篇：从 0 到 1 搭建 GopherAI 的 MCP 与工具调用模块

## 一、本篇要解决什么问题

当聊天模块和 RAG 模块都跑起来以后，AI 应用下一步很容易遇到一个问题：

- 如果模型需要实时天气、搜索结果、外部系统状态，它该怎么拿？
- 为什么不能让大模型“自己编一个结果”？
- 为什么工具调用不应该只是几段散落的 HTTP 请求？
- MCP 到底解决了什么，和普通接口调用有什么区别？

很多项目做到这里，会先在模型代码里直接写一个 `http.Get()`，或者在 Prompt 里硬塞一条“如果问天气就调用某接口”。这在 Demo 阶段很快，但很快就会出现下面这些问题：

1. 工具描述、工具执行、模型提示词全部耦在一处，越来越难维护。
2. 工具地址、超时、路径都是硬编码，换环境就要改代码。
3. 工具服务根本没有纳入主程序启动链路，前端能选“工具模型”，后端却不一定真正可用。
4. 模型到底什么时候该调用工具、什么时候直接回答，没有统一规则。
5. 工具失败时怎么降级、怎么回退、怎么继续回答，也没有清晰边界。

所以这一篇的目标，不是“再多接一个天气接口”，而是从 0 到 1 把 GopherAI 的 MCP 与工具调用链路搭成一个真正可维护、可配置、可扩展的工程模块。

这一轮在项目里的核心改造包括：

- 新增 `internal/modules/mcp/config.go`，统一 MCP 的运行配置、地址和默认值
- 新增 `internal/modules/mcp/catalog.go`，统一工具清单、规划 Prompt 和工具决策解析
- 新增 `internal/modules/mcp/client.go`，封装 MCP 客户端的初始化与工具调用
- 新增 `internal/modules/mcp/server.go`，承接天气工具注册与内部 MCP Server 启动
- 更新 `config/config.go` 和 `config/config.toml`，让 MCP 不再依赖硬编码
- 更新 `internal/bootstrap/bootstrap.go`，把 MCP Server 纳入主程序启动链路
- 更新 `common/aihelper/model.go`，让 `MCPModel` 从“内嵌 demo 逻辑”升级成真正依赖模块的实现

这一步的本质，是把“工具调用 demo”提升成“工具调用模块”。

## 二、先看整体流程：一条工具调用消息是怎么跑通的

### 1. 启动流程

```text
应用启动
-> 读取 config.toml 中的 mcpConfig
-> bootstrap.InitApplication()
-> StartInternalServer() 启动内部 MCP Server
-> 服务监听在 127.0.0.1:8081/mcp
-> Chat 模块恢复 AIHelper 时，MCPModel 已经有稳定运行边界
```

这里最重要的变化不是“多起了一个服务”，而是 MCP 终于从独立 demo 进入了主应用启动链路。

### 2. 一次普通工具调用流程

```text
用户选择模型 3（MCP 工具模型）并提问
-> MCPModel 拿到最后一条用户消息
-> 用 BuildPlannerPrompt() 让大模型先做“是否需要工具”的规划
-> 如果模型直接回答，则直接返回自然语言结果
-> 如果模型返回 useTool=true 的 JSON
-> ToolClient 初始化 MCP Client，并连接内部 /mcp 端点
-> 调用指定工具，比如 get_weather(city)
-> ToolClient 拿到工具文本结果
-> BuildAnswerPrompt() 把工具结果重新喂给大模型
-> 大模型生成最终面向用户的答案
```

这条链路体现出一个很重要的工程认知：

**工具调用不是让大模型自己去联网，而是让大模型先做规划，再让受控模块代替它执行工具。**

### 3. 流式工具调用流程

```text
用户发送流式问题
-> MCPModel 先同步做规划（是否用工具）
-> 如果不需要工具，直接流式输出回答
-> 如果需要工具，先调用 MCP Tool
-> 拿到工具结果后，再进入第二段流式回答
-> cb(chunk) 持续把最终答案写给前端
```

这里有一个细节很关键：当前实现里，“工具规划阶段”仍然是同步的，“最终回答阶段”才进入流式输出。因为工具调用链路里必须先知道：

- 要不要调工具
- 调哪个工具
- 工具返回了什么

只有这些信息确定了，最终自然语言回答才有办法稳定流出去。

### 4. 内部 MCP Server 的运行边界

```text
主业务服务监听 0.0.0.0:9090
内部 MCP 服务监听 127.0.0.1:8081
-> 外部用户访问业务 API
-> 业务模型通过本机 loopback 调用 MCP 工具端点
-> 工具能力不直接暴露给公网
```

这也是这次设计里非常值钱的一点：

**工具服务是主应用的内部能力，不是默认对外暴露的公共接口。**

## 三、项目里该怎么实现这部分能力

### 1. 为什么 MCP 配置必须独立出来

如果工具端点写死成：

```go
mcpBaseURL := "http://localhost:8081/mcp"
```

短期能跑，但后面很快就会有问题：

- 本地和容器环境地址可能不同
- 路径可能变化
- 端口可能冲突
- 超时时间不好统一管控
- 服务名、版本号也无法配置化

所以这次新增了 `mcpConfig`：

```toml
[mcpConfig]
enabled = true
host = "127.0.0.1"
port = 8081
path = "/mcp"
serverName = "gopherai-weather-server"
serverVersion = "1.0.0"
weatherBaseUrl = "https://wttr.in"
requestTimeoutSeconds = 10
```

这一层本质上是在做运行时治理：

- 地址统一管理
- 默认值统一兜底
- 环境切换更平滑
- 工具服务配置不再散落在模型代码里

### 2. 为什么要把 Prompt 规划和工具目录单独抽到 `catalog.go`

当前这轮只接了一个天气工具，但如果把工具描述直接写死在 `MCPModel` 里，后面一加工具就会开始乱：

- 模型 Prompt 一份
- Server 注册一份
- 文档里再写一份
- 解析逻辑里再判断一份

所以这次把这些内容统一收到了 `catalog.go`：

- `SupportedTools()` 负责定义工具目录
- `ToolCatalogPromptText()` 负责把工具列表转成 Prompt 文本
- `BuildPlannerPrompt()` 负责第一阶段“是否用工具”的规划提示词
- `BuildAnswerPrompt()` 负责第二阶段“拿着工具结果生成答案”的提示词
- `ParseToolDecision()` 负责把模型输出解析为结构化工具决策

这样做的价值是：

**工具定义、工具规划、工具解析开始有了单一事实来源。**

### 3. 为什么 MCPModel 要做成“两阶段调用”

这一轮没有直接走“模型原生 function calling”路线，而是先用了一条更容易观察和调试的两阶段链路：

1. 第一次让模型决定要不要用工具。
2. 如果要用工具，就交给 ToolClient 真正执行。
3. 再把工具结果回填给模型，生成最终回答。

也就是：

```text
Plan -> Execute Tool -> Final Answer
```

这样做的好处很明显：

- 规划阶段和执行阶段边界清晰
- 工具调用日志更容易排查
- 降级逻辑更容易写
- 以后切换到更多工具或更复杂编排时，结构更自然

当然它也有代价：

- 会多一次模型调用
- 工具规划质量依赖 Prompt 约束
- 还不是最强的原生 tool calling 形态

所以这不是终极形态，但很适合当前阶段把链路先搭稳。

### 4. 为什么 `ParseToolDecision()` 要求严格 JSON

当前的规划 Prompt 明确要求：

- 需要工具时只能返回 JSON
- 不要夹带 markdown 代码块和解释
- 不需要工具时直接返回自然语言

这件事看起来有点“严格”，但工程上非常有必要。因为如果允许模型随便输出：

- 一段自然语言 + 一个 JSON
- 一个代码块 + 一句解释
- 一个模糊的“我建议调用天气工具”

那你后端就会陷入非常脆弱的字符串猜测逻辑。

所以现在的解析策略是：

- 优先解析严格 JSON
- 兼容去掉 ```json 代码块
- 兼容旧字段 `isToolCall`
- 工具名必须在支持列表中

这比原来靠“包含某个关键词就当作工具调用”的方式稳很多。

### 5. 为什么要单独封装 `ToolClient`

如果让 `MCPModel` 自己负责：

- 创建 Transport
- 创建 Client
- 初始化协议版本
- 发送 `Initialize`
- 调用 `CallTool`
- 提取结果文本

那模型代码会很快变成一团运行时细节。

所以这次单独封装了 `ToolClient`，它负责：

- 惰性初始化 MCP Client
- 统一发起 `Initialize`
- 统一调用工具
- 统一把 `CallToolResult` 提取成文本
- 复用同一个客户端连接

这样 `MCPModel` 的关注点就回到了它该做的事情上：

- 规划
- 决策
- 调工具
- 回答

### 6. 为什么内部 MCP Server 要绑定 `127.0.0.1`

这次我没有把 MCP 直接挂到公网业务路由上，而是让它作为内部工具服务监听：

- `127.0.0.1:8081/mcp`

这样做有几个明显好处：

1. 工具调用面不会直接暴露给外部用户。
2. 业务 API 和工具协议边界更清晰。
3. 工具服务更像“应用内部基础设施”，而不是公共 HTTP 接口。
4. 后面加鉴权、审计、工具白名单时更自然。

这实际上是在做“最小暴露面”设计。

### 7. 为什么内部 MCP Server 先用 Stateless 模式

启动 MCP Server 时，这次使用的是：

- `WithStateLess(true)`

原因是当前工具场景还比较轻：

- 只有天气工具
- 没有复杂的长会话工具状态
- 主要需求是“工具可用”而不是“工具会话管理”

所以无状态模式更轻，也更适合内部 loopback 工具服务。

这也说明一件事：

**协议能力很多，但项目里不要一上来把所有复杂度都打开。**

### 8. 为什么工具失败时要回退，而不是整条链路直接崩掉

当前 `MCPModel` 的降级策略是：

- 如果工具规划失败，就直接返回第一次模型结果
- 如果工具执行失败，也直接回退到第一次模型结果
- 如果 MCP 被禁用，则直接走普通模型生成

这样做的价值不是“掩盖错误”，而是让系统在真实环境里更稳：

- 工具挂了，聊天不一定要完全不可用
- 用户至少能拿到一个自然语言结果
- 后端日志还能看到具体失败点

这就是很典型的工程思路：

**工具能力是增强层，不应该轻易把基础对话链路一起拖死。**

## 四、为什么这次要这样重构，而不是继续把天气工具留在 demo 目录里

### 1. 因为“存在代码”不等于“进入系统”

原来 `common/mcp` 下已经有独立 server/client/main，但它本质上更像一个演示目录：

- 有 demo server
- 有 demo client
- 有独立启动方式
- 但主程序启动时并不会管它
- 聊天主链路和它也没有真正模块化收口

所以这次真正解决的是：

**MCP 不再只是仓库里“有这份代码”，而是开始成为主应用运行的一部分。**

### 2. 因为工具调用不该散在模型实现细节里

如果工具调用逻辑一直嵌在 `common/aihelper/model.go` 里，后面只要再加两个工具，文件就会迅速变得难维护。抽成 `internal/modules/mcp` 以后，模型代码就只保留：

- 调规划 Prompt
- 解析结果
- 调工具客户端
- 做最终回答

这让职责边界清晰很多。

### 3. 因为工具能力本质上是协议与运行时问题

很多人一听 MCP，第一反应是“又多一个工具调用方式”。

但真正落到工程里，MCP 至少牵扯这几件事：

- 工具描述如何暴露
- 工具协议如何传输
- Client/Server 如何初始化
- 工具执行边界怎么控制
- 服务如何启动
- 地址和超时如何配置

所以它不是单纯的“写一个天气接口”，而是一个小型运行时模块。

## 五、实现中必须特别注意的坑

### 1. 工具目录和 Prompt 描述不能分裂

如果 `Server` 里注册的是一套工具，Prompt 里告诉模型的是另一套工具，模型就会规划出根本不存在的调用。工具清单一定要有统一来源。

### 2. 严格 JSON 不是吹毛求疵，而是为了后端可解析

Prompt 如果不收紧，后端很容易掉回脆弱的字符串猜测。

### 3. 内部工具端点不要默认暴露公网

否则你等于平白新增了一套可以被外部直接调用的协议面。工具服务应该默认走内网或 loopback。

### 4. 工具失败不能简单等于整个聊天失败

真实系统里，增强链路失败时要考虑优雅回退，否则一个天气接口抖一下，整个聊天模型都报错，用户体验会非常差。

### 5. Prompt 规划方案虽然直观，但不是最终形态

当前两阶段方案很好调试，但它不是最强的工具调用形态。后面如果工具越来越多，可能要继续升级到：

- 原生 function calling
- 更强的工具路由器
- 多工具串联执行
- 工具结果结构化注入

### 6. 工具调用一定要传递 `context`

这样用户取消请求、超时、链路中断时，工具 HTTP 请求也能尽快停下来，避免浪费资源。

### 7. ToolClient 的初始化和复用要考虑并发

如果每次都重新建 client、重新 initialize，会增加不必要开销；如果完全不加保护，又可能出现并发初始化问题。这也是为什么这里单独做了客户端封装。

## 六、围绕本篇内容，需要进一步掌握哪些技术基础

### 1. MCP 到底是什么

MCP 是 Model Context Protocol，用来标准化模型与外部能力之间的交互方式。它不只关心“怎么调用工具”，还关心：

- 工具如何被发现
- 请求和响应如何组织
- Client 与 Server 如何协商能力
- 除了 tools 之外，resources、prompts 等能力如何表达

### 2. MCP 和普通 HTTP API 有什么区别

HTTP API 只解决“请求能发过去”，MCP 解决的是“模型侧工具生态怎么标准化”。

简单说：

- HTTP 更像传输手段
- MCP 更像工具协议约定

所以 MCP 可以跑在 HTTP 上，但它表达的层次比“写几个接口”更高。

### 3. 为什么工具调用通常不是一次模型调用就结束

因为真实系统里通常至少有两层：

- 一层决定要不要用工具
- 一层拿着工具结果做最终回答

工具结果本身通常不是最终用户想看的内容，模型还需要把它组织成人类更容易理解的自然语言。

### 4. 什么是 JSON-RPC

MCP 底层消息结构是基于 JSON-RPC 的，所以理解这些概念很重要：

- method
- params
- result
- error
- request / response / notification

这能帮助你理解为什么 MCP 不是普通 REST 风格接口。

### 5. 工具调用为什么是安全边界问题

因为一旦模型能够间接驱动外部能力，就意味着：

- 它可能读数据
- 可能发请求
- 可能访问内部服务
- 未来甚至可能执行带副作用动作

所以工具能力一定要考虑最小暴露面、参数校验、白名单和审计。

## 八、提问 + 标准答案

下面这些问题既适合你做项目复盘，也很贴近国内大中厂围绕工具调用、协议设计和 AI 应用工程化的常见追问。

### 问题 1：MCP 是什么？它解决了什么问题？

**标准答案：**

MCP 是 Model Context Protocol，目的是标准化模型与外部能力之间的交互方式。它不只是定义“怎么调用工具”，还定义了能力发现、消息协议、客户端和服务端协商方式。相比直接写散乱 HTTP 接口，MCP 更适合把工具、资源、提示词等能力放进统一协议里管理。

### 问题 2：MCP 和普通 HTTP API 有什么区别？

**标准答案：**

HTTP API 更像底层传输手段，只解决“请求发过去、结果拿回来”；MCP 是更高一层的协议约定，关注的是模型如何发现工具、如何声明能力、如何标准化调用。可以理解为 HTTP 解决传输，MCP 解决模型工具生态的标准化。

### 问题 3：为什么不直接在模型代码里写 `http.Get()` 调天气接口？

**标准答案：**

因为那样很快就会把工具描述、工具执行、超时、地址、降级、日志全部耦进模型实现里，后续加工具会越来越乱。把它抽成 MCP 模块后，模型负责规划和回答，工具执行交给独立客户端和服务端，更符合工程边界。

### 问题 4：为什么这次要把 MCP 配置独立出来？

**标准答案：**

因为工具服务的 host、port、path、超时、上游地址都属于运行时配置，不应该硬编码在模型里。配置独立出来后，环境切换、容器部署和问题排查都会更稳定。

### 问题 5：为什么内部 MCP Server 要纳入 bootstrap，而不是手动单独启动？

**标准答案：**

因为只有进入主应用启动链路，这个能力才真正算“系统的一部分”。否则仓库里虽然有 MCP 代码，但主应用启动时不保证它可用，前端选了工具模型也未必真的能执行工具。

### 问题 6：为什么这次采用的是“两阶段调用”而不是一步到位？

**标准答案：**

因为当前阶段更看重链路清晰和可调试性。先让模型判断是否要用工具，再由系统执行工具，最后再让模型基于工具结果回答，这样更容易观察问题出在规划、执行还是回答阶段。代价是多一次模型调用，但工程上更稳。

### 问题 7：为什么规划阶段要求模型严格返回 JSON？

**标准答案：**

因为后端需要稳定地解析工具决策。如果允许模型返回“自然语言 + JSON + 解释”，解析逻辑会非常脆弱。严格 JSON 是为了让后端有可靠的结构化输入。

### 问题 8：为什么要兼容去掉 ```json 代码块和旧字段 `isToolCall`？

**标准答案：**

因为真实系统里模型输出不一定始终完全稳定，兼容这类常见格式可以让链路更稳。同时兼容旧字段也能降低重构期间对历史逻辑的冲击，体现渐进式演进能力。

### 问题 9：为什么 ToolClient 要做成单独封装？

**标准答案：**

因为 MCP Client 的创建、Initialize、CallTool、结果提取这些细节本质上是运行时基础设施，不应该混在模型业务里。单独封装以后，模型实现会更聚焦，也更容易测试和复用。

### 问题 10：为什么内部 MCP Server 绑定在 `127.0.0.1`，而不是对外公开？

**标准答案：**

因为当前工具服务本质上是应用内部基础设施，不是默认给公网开放的协议面。绑定 loopback 可以减少攻击面，也更符合最小暴露面原则。

### 问题 11：为什么当前 MCP Server 先使用 Stateless 模式？

**标准答案：**

因为当前场景主要是轻量工具查询，没有复杂的服务端会话状态。无状态模式更简单、运维成本更低，也足够满足当前内部工具调用需求。

### 问题 12：为什么工具失败时要回退到第一次模型结果？

**标准答案：**

因为工具能力是增强层，不应该轻易把基础聊天一起拖死。工具失败后回退到自然语言结果，可以保证系统仍然可用，同时把错误留在日志里做排查。

### 问题 13：为什么不能让模型直接“自己决定并自己执行”工具？

**标准答案：**

因为模型本身不应该直接拥有任意外部执行能力。正确做法是模型只负责规划，真正的执行由受控系统代劳，这样才有权限边界、参数校验、日志和回退能力。

### 问题 14：MCP 和 OpenAI function calling 是什么关系？

**标准答案：**

它们都在解决“模型如何和外部能力交互”的问题，但层次不完全一样。function calling 更像某个模型供应商提供的调用机制，MCP 更强调跨 Host / Client / Server 的协议标准化。可以把 MCP 理解成更通用的协议层。

### 问题 15：为什么工具结果还要再交给模型生成最终答案？

**标准答案：**

因为工具结果往往是结构化或半结构化数据，本身不一定适合直接给用户看。模型需要把工具结果和用户问题重新组织成自然语言回答，这样体验更好，也更符合对话场景。

### 问题 16：如果以后工具越来越多，这一版设计还能扩吗？

**标准答案：**

能扩。因为当前已经把工具目录、Prompt、解析逻辑、客户端和服务端都抽成了模块。后面加更多工具、加白名单、加权限、加审计，都有清晰挂点，不需要继续把逻辑塞回模型文件里。

### 问题 17：如果面试官问你“为什么不用直接 REST，而要引入 MCP”，你怎么回答？

**标准答案：**

我会说 REST 当然可以解决单一工具调用，但项目一旦进入多工具、多客户端、多运行环境场景，只靠散乱 REST 很快就会缺少统一的工具发现、调用协议和扩展边界。引入 MCP 的价值在于让工具能力进入统一标准，而不是每次都重新发明一套约定。

### 问题 18：工具调用链路里最需要关注哪些安全问题？

**标准答案：**

重点包括：最小暴露面、参数校验、上游访问控制、超时和取消、日志审计、工具白名单，以及未来涉及副作用操作时的权限控制。工具能力本质上就是一层执行边界，安全一定不能后补。

### 问题 19：如果这条链路以后要做可观测性，你会怎么补？

**标准答案：**

我会补这些指标和日志：规划耗时、工具调用耗时、工具成功率、回退次数、模型第二次生成耗时、工具参数摘要、错误类型分布。这样才能知道问题出在规划、工具、上游还是最终回答阶段。

### 问题 20：如果让你继续优化这块，下一步你会补什么？

**标准答案：**

我会优先补：

- 多工具注册与白名单
- 更细粒度的工具参数校验
- 工具调用审计日志
- 原生 function calling / tool calling 升级
- 工具结果结构化传递
- 更完善的超时、重试和熔断策略
- 会话级工具配置与权限隔离

这说明项目会从“能调用一个工具”继续演进到“能稳定、可控地管理工具系统”。
