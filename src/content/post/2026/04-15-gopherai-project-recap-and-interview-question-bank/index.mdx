# 第八篇：GopherAI 项目复盘与大厂面试题单

## 一、本篇要解决什么问题

到 Phase 8，GopherAI 的重构已经不再缺“功能模块”，而是开始进入一个更重要的问题：

- 如果面试官让你用 30 秒介绍这个项目，你怎么说？
- 如果面试官继续追问架构、链路、难点、取舍和短板，你能不能讲深？
- 如果对方从 Go、JWT、RAG、MCP、多模态、可观测性、前端工程化这些方向分别发问，你的回答能不能既成体系、又有项目落点？

很多同学做项目时，代码其实已经写了不少，但一到复盘时就会出现两个问题：

1. 讲不清项目全貌，只能零碎讲某个接口或某个页面。
2. 能讲“做了什么”，但讲不清“为什么这样做、有哪些权衡、还存在哪些不足”。

所以这一篇的目标不是继续加功能，而是把 GopherAI 重构成一份真正可用于大中厂面试表达的项目资产。

## 二、先学会怎么介绍这个项目

### 1. 30 秒版本

GopherAI 是一个基于 Go + Vue 的全栈 AI 应用，我把它从传统的 `controller/service/dao/router` 分层结构，逐步重构成按业务能力组织的模块化项目。核心能力包括认证与启动、会话聊天、RAG 知识库、MCP 工具调用、多模态、可观测性和前端工作流，重点解决了 AI 应用里会话一致性、知识入库一致性、工具协议接入、运行时观测和前端工程边界这些问题。

### 2. 3 分钟版本

如果面试官愿意听长一点，可以按下面这个顺序讲：

1. **项目定位**：GopherAI 是一个面向 AI 助手场景的全栈应用，支持用户注册登录、聊天、多轮会话、知识库上传、图像识别、语音合成和工具调用。
2. **重构背景**：项目原来主要按技术层分目录，随着功能增多，认证、聊天、RAG、工具调用这些能力被分散在很多目录里，维护成本越来越高。
3. **重构目标**：我没有一次性推翻重写，而是采取“新模块落地 + 旧接口兼容”的方式，逐步把核心能力收口到 `internal/modules`，把风险控制在可演进范围内。
4. **关键模块**：我先后把认证与启动、聊天与会话、知识入库、MCP、多模态、可观测性和前端体验收成模块，并分别补了文档、验证和博客。
5. **核心收益**：项目从“功能能跑”变成“边界清晰、链路可讲、问题可定位、面试可复盘”的状态。

### 3. 一句话讲出你的价值点

如果面试官继续问“这个项目里最体现你能力的部分是什么”，建议优先讲这三点：

- 我做的不是堆功能，而是把项目往工程化和模块化方向推进。
- 我不仅改了代码，还同步沉淀了设计理由、坑点、验证方式和面试表达。
- 我关注的不只是单个接口，而是整条 AI 应用链路如何稳定协同。

## 三、整套项目可以怎么拆给面试官听

### 1. 认证与启动层

这一层解决两个问题：谁在访问系统，系统如何按顺序启动。

核心点：

- 认证能力单独收口，拆分了 token 生成解析、中间件鉴权和上下文读取。
- 启动流程从 `main.go` 中抽离，统一初始化 MySQL、会话恢复、Redis、RabbitMQ 和内部 MCP 服务。
- 保留旧调用面兼容，降低迁移风险。

面试里这层最容易被问到：

- 为什么用 JWT，不直接用 Session？
- JWT 为什么必须配 HTTPS？
- 为什么启动层也需要模块化？

### 2. 会话与聊天层

这一层是 AI 应用的主链路，解决“消息如何进入系统并形成可持续会话”。

核心点：

- 会话列表从数据库读取，不再只靠内存态。
- 历史消息按显式角色组织，不再依赖奇偶位置猜测身份。
- 普通发送和流式发送都支持，并把 `request context` 贯穿到聊天链路。
- 新会话创建、消息落库、启动恢复都做了兼容设计。

面试里这层最容易被问到：

- 为什么聊天不能只理解成“调一次模型接口”？
- 流式输出和普通输出的差异是什么？
- 为什么要把 request context 传到 service 层？

### 3. 知识库与 RAG 层

这一层解决“用户文件如何真正变成可检索知识”。

核心点：

- 引入知识文件元数据表，而不是只扫 `uploads/<username>` 目录。
- 用稳定的 `indexKey` 关联文件、索引和元数据，避免文件名变化带来的歧义。
- 把上传成功和入库成功区分开，开始建立一致性意识。

面试里这层最容易被问到：

- 为什么文件上传成功不等于知识库可用？
- 为什么 RAG 需要元数据表？
- 多用户场景下知识隔离在哪一层做？

### 4. MCP 与工具调用层

这一层解决“模型如何以可扩展的方式使用工具”。

核心点：

- 把工具目录、调用规划、协议边界和内部 MCP Server 独立成模块。
- 模型不再硬编码散落的 HTTP 请求，而是走“规划 -> 调工具 -> 返回结果”的链路。
- 配置化管理工具服务地址，避免实现与环境强耦合。

面试里这层最容易被问到：

- MCP 到底解决了什么问题？
- MCP 和直接调 HTTP 接口有什么区别？
- 工具调用为什么不能只靠 Prompt 硬写规则？

### 5. 多模态层

这一层解决“图像识别和 TTS 如何真正进入工程主链路”。

核心点：

- 图像识别改为复用共享识别器，而不是每次请求重建推理环境。
- TTS 引入 HTTP Client 复用和第三方 token 缓存，减少无意义开销。
- 上传校验补齐了扩展名、MIME、解码三层检查。

面试里这层最容易被问到：

- 为什么图像识别不能每次请求都重新加载模型？
- 第三方平台 access token 为什么要缓存？
- 前端校验和后端校验分别承担什么职责？

### 6. 评估与可观测性层

这一层解决“系统运行得怎么样、出问题时如何定位”。

核心点：

- 做了请求级观测、业务级操作记录和启动级记录。
- 用 `request id` 和统一业务码写法把观测串起来。
- 提供总览接口，先把“看得见系统状态”做起来，再逐步接更完整的 metrics/tracing。

面试里这层最容易被问到：

- 为什么先做基础观测，而不是一开始就全量接 Prometheus 和 OpenTelemetry？
- `request id` 和 `trace id` 的区别是什么？
- 为什么 `HTTP 200` 不能等于“业务成功”？

### 7. 前端体验与工程化层

这一层解决“前端如何从 Demo 页演进成产品前台”。

核心点：

- 区分公开页与受保护页面，引入统一 App Shell。
- API 层统一 token、401、业务码和错误提示处理。
- 聊天页重构成工作流页面，支持会话切换、流式输出、知识上传和 TTS。
- 图像识别页补齐预览、本地历史和前端校验。

面试里这层最容易被问到：

- 为什么路由守卫不是安全边界？
- 为什么要统一封装 API 层？
- 为什么大模型输出渲染要考虑 XSS？

## 四、这套项目里最值得讲的工程亮点

如果面试时间有限，建议优先抓这五个亮点：

### 1. 不是推翻重写，而是兼容式模块化迁移

这是一个很有工程味道的点。很多遗留项目不会允许你一口气全部重写，所以“新模块逐步落地 + 旧接口继续可用”本身就是能力。

### 2. 把 AI 应用拆成真正的业务链路，而不是单个模型调用

从认证、会话、知识库、工具调用到多模态和前端体验，整个项目强调的是链路协同。这比单纯讲“我接了一个大模型 API”更有说服力。

### 3. 强调一致性和边界意识

比如：

- 上传成功不等于知识库可用。
- HTTP 成功不等于业务成功。
- 路由守卫不等于安全边界。
- 前端校验不等于后端校验。

这类表达非常适合面试，因为它能直接体现你理解系统边界，而不是只会堆功能。

### 4. 有性能与资源复用意识

比如共享图像识别器、缓存 TTS token、路由懒加载、统一 HTTP Client、会话冷启动恢复等，这些都能说明你已经在考虑真实运行成本。

### 5. 不只写代码，还做验证和知识沉淀

每个阶段都有博客、文档和验证，这非常像真实团队里的“交付闭环”。对面试官来说，这往往比“我一个晚上写了多少行代码”更值钱。

## 五、如果面试官问“你遇到的最大难点是什么”，可以怎么答

这里建议不要答得太虚。可以从下面三个角度选一个最贴近你的真实经历的版本：

### 1. 兼容式重构的难点

难点不是“我不会写新代码”，而是“我要在旧接口不破坏的前提下，把核心能力逐步迁入新模块”。这要求我在抽象边界时既要看未来结构，也要兼顾当前调用面。

### 2. AI 场景不是单链路问题，而是多链路协同问题

比如聊天这条线，表面只是发消息，但实际上会牵扯会话状态、历史消息、流式输出、知识检索、工具调用、落库、前端渲染等多个环节。真正难的是把这些环节组织成稳定系统，而不是只让某个点能跑。

### 3. 问题定位和可讲性

很多项目的难点不只是“把问题修掉”，而是“知道问题在哪、为什么出现、以后怎么避免、怎么向别人解释”。这也是为什么后面我会把可观测性和文档沉淀纳入重构的一部分。

## 六、如果面试官问“还有哪些不足”，你应该怎么答

这个问题一定不要答成“我这个项目已经很完美了”。更成熟的回答应该是：

1. **RAG 侧还有进一步空间**：目前已经建立了知识文件元数据和索引一致性，但在召回质量评估、切块策略对比、embedding 效果分析上还可以继续深化。
2. **工具调用侧还可以加权限与治理**：MCP 已经接入主链路，但未来还可以补工具级权限、超时重试、熔断和更细的审计记录。
3. **观测体系还可以继续升级**：当前已经有 overview 和操作记录，下一步可以继续接标准 metrics、tracing 和 dashboard。
4. **前端还可以继续做状态治理和包体优化**：目前已经完成页面边界和 API 分层，但集中状态管理、依赖裁剪和更细的性能治理仍有空间。
5. **认证层还可以完善会话失效策略**：例如引入 refresh token、黑名单或更细粒度的登录态管理。

这种回答的好处是：既承认不足，又能体现你知道下一步该怎么做。

## 七、建议你这样准备项目面试

### 1. 先准备三套版本

- 30 秒版：一句话 + 三个关键词
- 3 分钟版：背景、目标、做法、结果
- 深挖版：按模块展开，能接住追问

### 2. 把问题分成四类去准备

- 架构类：为什么这样拆模块
- 技术类：JWT、Go、Gin、Vue、SSE、RAG、MCP、多模态
- 工程类：兼容式重构、可观测性、配置化、性能优化
- 反思类：不足、取舍、下次怎么做

### 3. 回答时尽量带项目落点

不要只背概念。比如不是只说“JWT 是无状态的”，而是说“因为 GopherAI 是前后端分离场景，并且后续有工具调用和多模块链路，所以我优先用了 JWT，但也明确知道它在失效控制上不如 Session 灵活”。

## 九、项目面试题单：提问 + 标准答案

### 1. 请你先整体介绍一下 GopherAI 这个项目。

**标准答案：**
GopherAI 是一个基于 Go 和 Vue 的全栈 AI 应用，支持用户认证、会话聊天、知识库上传、RAG 检索、MCP 工具调用、图像识别、语音合成和可观测性总览。我做的核心工作不是单点加功能，而是把它从传统按层分目录的结构，逐步重构成按业务能力组织的模块化项目，并同步完成了验证、文档和博客沉淀。

### 2. 为什么你要重构这个项目，而不是继续往原结构上加功能？

**标准答案：**
因为功能变多以后，认证、聊天、RAG、工具调用这些能力会散落在 controller、service、dao、router 等多个目录里，改一处往往要来回跳很多层。继续堆功能只会让维护成本持续上升。重构的目标是把能力边界收回来，让项目更适合持续迭代。

### 3. 为什么按功能模块组织，比按 controller/service/dao 分层更适合这个项目？

**标准答案：**
按层分目录适合项目早期，但在 AI 应用这种链路复杂的场景里，一个完整功能往往横跨多层。按功能模块组织能把认证、聊天、知识库、工具调用这些能力收在一起，更方便维护、扩展和定位问题。

### 4. 你为什么用了 JWT，而不是 Session？

**标准答案：**
因为 GopherAI 是前后端分离场景，JWT 更方便通过 Header 透传身份，也更适合后续多模块、多服务协同。但我不会说 JWT 一定优于 Session，因为如果业务更强调服务端强控登录态和立即失效，Session 反而更合适。

### 5. JWT 为什么还要配 HTTPS？

**标准答案：**
JWT 的签名只能保证内容没有被篡改，不能防止 token 在传输过程被窃取。HTTPS 解决的是链路窃听和中间人问题，所以二者解决的是不同层面的安全问题，必须配合使用。

### 6. 你这里有没有考虑 JWT 失效和踢人下线的问题？

**标准答案：**
当前方案主要是基础 JWT 登录态维持，更适合先把前后端分离场景跑通。如果业务要进一步做精细会话控制，我会补 refresh token、黑名单或 Redis 侧状态控制，让登录态具备更强的主动失效能力。

### 7. 为什么启动流程也值得单独重构？

**标准答案：**
因为随着组件变多，启动本身就是一条关键链路。把初始化逻辑堆在 `main.go` 里会导致顺序混乱、依赖关系不清，也不利于后续做启动耗时统计和故障定位。所以我把它抽成 bootstrap，统一管理启动步骤。

### 8. 聊天模块为什么不能理解成“前端发问题，后端调模型，返回答案”？

**标准答案：**
因为真实聊天链路不只是模型调用，还会牵扯会话归属、历史消息、上下文拼接、流式输出、RAG 检索、工具调用、落库和恢复等问题。真正难的是把这条链路做成可维护的系统，而不是只把某次回答跑通。

### 9. 会话为什么必须持久化？只存在内存里不行吗？

**标准答案：**
只存内存的问题是重启后丢失、无法多实例共享、也很难保证用户会话隔离。持久化以后，系统才有机会实现历史回放、启动恢复和一致性的会话管理。

### 10. 流式输出在工程上最大的难点是什么？

**标准答案：**
不是“把内容一段段返回”这么简单，而是要处理分片拼接、结束标记、中途断流、前端占位消息和新会话创建后的 sessionId 回填。流式输出体验更好，但对链路组织要求也更高。

### 11. 为什么你把 `request context` 传进了聊天链路？

**标准答案：**
因为 request context 能把取消信号、超时和 request 级上下文从 HTTP 层传到 service 层，让后续模型调用、工具调用和异步流程更容易协同，也更符合 Go 的工程实践。

### 12. 为什么文件上传成功不等于知识库可用？

**标准答案：**
因为上传只是文件进了服务端，后面还要经过切块、索引、元数据记录和查询链路接入。只要其中一个环节失败，就不能把它当成“知识库已经可检索”。

### 13. 为什么你要给知识文件建元数据表，而不是只扫目录？

**标准答案：**
只扫目录拿不到足够的业务状态，比如当前文件是否生效、对应的索引键是什么、替换时该如何处理旧索引。元数据表能把文件、索引和业务状态稳定关联起来。

### 14. 你这里的 RAG 设计里，最关键的稳定性点是什么？

**标准答案：**
是把知识文件元数据、稳定 `indexKey` 和实际索引构建过程串起来，开始建立“上传、入库、检索”三者一致性的意识。这样后续才能继续扩展成更复杂的知识库形态。

### 15. MCP 在这个项目里主要解决什么问题？

**标准答案：**
MCP 解决的是模型与工具之间的标准化接入问题。它让工具调用从“散落的临时接口”变成“可规划、可注册、可扩展的协议层能力”，这样模型接新工具时更容易收口。

### 16. MCP 和直接调 HTTP 接口的区别是什么？

**标准答案：**
直接调 HTTP 也能拿到结果，但调用方式、参数结构和描述信息通常都很散。MCP 更强调标准化描述、能力发现和统一调用边界，更适合工具规模变大之后的治理和扩展。

### 17. 为什么图像识别不能每次请求都重新加载模型？

**标准答案：**
因为模型加载本身往往很重，如果每次请求都重新初始化，延迟和资源开销都会很高。在真实工程里，更合理的做法是复用共享识别器，并做好并发保护。

### 18. 第三方平台的 access token 为什么要缓存？

**标准答案：**
因为频繁申请 token 既浪费时间，也会增加第三方接口压力和限流风险。缓存 access token 能降低额外开销，提高系统稳定性。

### 19. 你为什么先做基础可观测性，而不是直接全量接 Prometheus 和 OpenTelemetry？

**标准答案：**
因为当时更紧迫的问题是先把请求、业务操作和启动链路看清楚，建立最小可用的观测闭环。等这些信息被稳定沉淀下来，再接标准 metrics 和 tracing，会更有方向，也更贴近当前项目阶段。

### 20. `request id` 和 `trace id` 有什么区别？

**标准答案：**
`request id` 更像单次请求的唯一标识，适合日志串联和快速定位；`trace id` 更强调分布式调用链上的整体追踪，通常会配合 span 一起使用。当前项目先建立了 request id 体系，后续可以继续向 tracing 演进。

### 21. 为什么说 `HTTP 200` 不等于业务成功？

**标准答案：**
因为很多接口会返回 `HTTP 200`，但 body 里的业务码仍然表示失败。如果只看 HTTP 状态，前端会把很多业务失败误判成成功，这对用户提示、埋点和问题定位都不利。

### 22. 为什么前端路由守卫不能算真正的权限控制？

**标准答案：**
因为路由守卫只发生在前端，最多做到页面跳转控制和体验优化。真正的权限控制必须由后端接口来做，用户完全可以绕过前端直接请求 API。

### 23. 你为什么要单独做统一 API 层？

**标准答案：**
因为 token 注入、401 处理、业务码拆分、错误提示和流式请求解析都属于横切逻辑。如果把这些逻辑散在页面里，后面会非常难维护。统一 API 层可以让页面更聚焦自己的工作流。

### 24. 大模型输出为什么会牵扯 XSS 问题？

**标准答案：**
因为模型输出本质上也是不可信输入。如果前端直接把它当 HTML 插进页面，就存在注入风险。更稳妥的做法是默认按文本处理，必要时先转义，再在受控范围内做 Markdown 渲染。

### 25. 这个项目里你最看重的工程能力体现是什么？

**标准答案：**
我最看重的是边界意识和兼容式重构能力。这个项目不是一个新项目白纸起手，而是在旧结构之上逐步把核心链路收成模块，还要尽量不破坏原有接口和前端调用面。

### 26. 如果再做一轮，你最想优先补哪一块？

**标准答案：**
我会优先补三块：RAG 召回质量与评估体系、工具调用治理能力、标准化 metrics/tracing。因为这三块会直接影响 AI 应用的可用性、可观测性和可扩展性。

### 27. 如果以后把这个项目继续演进成多服务，你会怎么拆？

**标准答案：**
我会优先按业务能力拆，比如认证、聊天编排、知识库处理、工具调用、多模态和观测服务，而不是简单按技术层拆。因为当前重构本身已经按能力边界做了准备，后续拆服务时会更自然。

### 28. 这个项目最适合怎样在简历上写？

**标准答案：**
不要写成“做了一个聊天网站”，而要写成“将一个 Go + Vue 的 AI 应用逐步重构为模块化工程，覆盖认证、会话聊天、RAG、MCP、多模态、可观测性与前端工作流，并形成完整技术沉淀与面试复盘”。这样更能体现项目深度。
