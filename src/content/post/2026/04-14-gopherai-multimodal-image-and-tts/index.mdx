## 一、本篇要解决什么问题
当认证、会话、RAG、MCP 都已经落下来以后，GopherAI 很自然会继续往多模态走：

- 用户希望上传一张图片，让系统识别类别。
- 用户希望把大模型生成的答案转成语音，直接播放。
- 前端希望继续维持统一的聊天体验，而不是突然冒出两套割裂的小工具。

很多项目做到这一步，最容易出现一种“看起来能跑、实际上很脆”的实现方式：

- 图像识别每次请求都重新加载模型。
- TTS 每次请求都重新向第三方平台申请 access token。
- 控制器里既做参数校验，又做 HTTP 调用，又做结果转换。
- 模型路径、音色、超时、上传大小全都写死在代码里。

这种写法 Demo 阶段确实快，但一旦用户量上来、请求并发增加、配置需要切环境、第三方接口开始抖动，就会迅速暴露出工程问题。

所以这一篇的目标不是“多接两个 API”，而是把 GopherAI 的多模态能力真正做成一个可维护的业务模块。当前这一轮重构主要完成了下面几件事：

- 新增 `internal/modules/multimodal/config.go`，统一图像识别与 TTS 的运行配置。
- 新增 `internal/modules/multimodal/image.go`，把上传校验、模型复用、内容探测和推理入口收口到同一个模块。
- 新增 `internal/modules/multimodal/tts.go`，把百度 TTS 的 token 缓存、HTTP Client、任务创建和任务查询统一封装。
- 更新 `common/image/image_recognizer.go`，为共享 ONNX 推理器补上并发保护。
- 新增 `common/image/image_recognizer_windows.go`，为 Windows 本地开发提供编译期兼容 stub，不影响 Linux 正式运行。
- 更新 `service/image/image.go`、`common/tts/tts.go`、`controller/image/image.go`、`controller/tts/tts.go`，把旧调用面平滑接到新模块上。

这一阶段的本质，是把“图像识别 + 语音合成”从零散接口升级成统一的多模态运行时。

## 二、先看整体流程：一条多模态请求是怎么跑通的
### 1. 图像识别请求流程
```text
前端上传图片
-> Gin controller 接收 multipart/form-data
-> multimodal.ImageService 校验扩展名、文件大小和 MIME 类型
-> 服务端再做一次图片解码检查，避免伪造文件通过
-> 获取全局复用的 ImageRecognizer
-> 复用 ONNX Runtime session 完成推理
-> 返回类别名称给前端页面
```
这条链路里最关键的点不是“能识别”，而是“不能每来一次请求就重建一次模型运行时”。

### 2. TTS 创建与轮询流程
```text
前端提交文本内容
-> TTS controller 调用 multimodal.SpeechService.CreateTask()
-> 服务端从缓存里拿百度 access token
-> 如果 token 不存在或即将过期，再向百度 OAuth 接口申请新 token
-> 调用百度 TTS create 接口创建异步任务
-> 返回 task_id 给前端
-> 前端轮询 query 接口
-> SpeechService.QueryTask() 查询任务状态
-> 状态成功后返回 speech_url
-> 前端拿到音频地址后播放
```
这里的重点是：当前接入的百度 TTS 不是“同步返回音频字节”的模型，而是“先创建任务、再查询结果”的异步任务模型。

### 3. 多模态运行时的初始化与复用流程
```text
应用启动
-> 读取 imageRecognitionConfig 与 ttsConfig
-> 图片模型路径、输入尺寸、上传限制被统一配置化
-> TTS 的 baseURL、超时、音色、语速、音量被统一配置化
-> 首次图像请求时惰性创建 ImageRecognizer
-> 首次 TTS 请求时惰性创建带 timeout 的 HTTP Client，并缓存 access token
-> 后续请求直接复用运行时对象
```
这一步很像我们在会话模块里做的 AIHelper 管理：真正昂贵的运行时对象应该尽量复用，而不是每次请求都重新造一份。

## 三、项目里该怎么实现这部分能力
### 1. 先把运行参数收进统一配置
这一轮先把多模态相关参数从代码中抽出来，落到 `config/config.go` 和 `config/config.toml`：

- 图像识别：`modelPath`、`labelPath`、`inputHeight`、`inputWidth`、`maxUploadSizeMB`
- TTS：`baseUrl`、`requestTimeoutSeconds`、`tokenBufferSeconds`、`format`、`voice`、`lang`、`speed`、`pitch`、`volume`、`enableSubtitle`

这样做有三个直接价值：

- 模型文件路径不再写死在 `service/image/image.go` 里。
- 百度 TTS 参数调优不需要反复改代码再重新部署。
- 本地、测试、容器环境的差异可以通过配置消化，而不是通过 if/else 消化。

### 2. 图像识别服务为什么要做成单例复用
旧实现里最大的性能问题，是每次 `RecognizeImage` 都会：

1. 创建新的 ONNX Runtime session。
2. 重新读取 label 文件。
3. 请求结束后再把 session 关闭。

如果把这条链路放到真实环境里，会出现非常明显的问题：

- 每次请求的启动成本很高。
- ONNX Runtime 初始化会放大响应延迟。
- 并发一上来，CPU 和内存抖动会很明显。

这次重构后，`internal/modules/multimodal/image.go` 引入了全局 `ImageService`，它会按配置生成一个 recognizer key。只要模型路径和输入尺寸没变，就直接复用同一个 `ImageRecognizer`。只有配置变化时，才会关闭旧 recognizer 并重建新的实例。

这就是典型的“昂贵运行时对象复用”思路。

### 3. 为什么共享推理器必须补并发锁
如果图像识别器从“每请求创建一个”改成“全局共享一个”，新的问题就会马上出现：共享对象是否线程安全。

`common/image/image_recognizer.go` 里持有这些共享资源：

- ONNX session
- 输入 tensor
- 输出 tensor

而 `PredictFromImage()` 在推理前后会修改输入 tensor，并读取输出 tensor。如果多个请求并发进入而不加锁，结果可能包括：

- 输入数据互相覆盖
- 输出结果串台
- session 运行状态冲突

所以这一轮专门给 `ImageRecognizer` 加了 `sync.Mutex`，并把共享 tensor 的写入、`session.Run()`、输出读取放进同一段临界区里。这个改动很小，但工程价值非常大，因为它决定了“全局复用”到底是性能优化，还是并发 bug。

### 4. 服务端为什么不能只看文件后缀
前端页面通常会写 `accept="image/*"`，但这只是用户体验提示，不是安全边界。

如果我们只看文件名后缀，比如：

- `evil.jpg`
- `note.png`

并不能保证它真的是图片。用户完全可以把任意文件改后缀上传。

这次服务端做了三层校验：

1. 看扩展名是否在允许列表里。
2. 用 `http.DetectContentType` 做 MIME 嗅探。
3. 再用 `image.DecodeConfig` 真正尝试解析图片头。

只有三层都过了，才会进入推理阶段。这也是上传类接口非常典型的工程防线设计。

### 5. 为什么 TTS 服务必须做 token 缓存
百度 TTS 接口需要 access token。旧实现里 `CreateTTS()` 和 `QueryTTSFull()` 都会每次调用 `GetAccessToken()`，也就是：

- 创建任务先请求一次 token
- 查询结果再请求一次 token
- 同一个用户一次播放行为可能要多次触发 token 请求

这会带来三个问题：

- 多余的网络开销
- 第三方接口限流风险更高
- 整体时延被平白放大

所以这次在 `internal/modules/multimodal/tts.go` 里加了 token 缓存逻辑：

- token 未过期且离过期还有 buffer，就直接复用。
- 只有 token 缺失或即将过期，才重新申请。
- 过期时间来自 OAuth 返回的 `expires_in`，而不是拍脑袋写死。

这是接第三方平台时非常重要的基本功：认证凭证本身也是运行时资源，不能当成普通字符串来用。

### 6. 为什么 TTS 这里用异步任务，而不是同步直接返回音频
这是一个非常适合写进博客、也很适合面试回答的点。

并不是所有 TTS 服务都适合同步返回音频。有些平台会采用“创建任务 -> 查询结果”的异步模型，原因通常有：

- 长文本合成时间不可控
- 后端需要调度更重的合成资源
- 平台希望隔离创建请求与结果下载
- 查询接口能让前端更好处理加载状态与失败重试

在 GopherAI 当前实现里，前端先调用 `/AI/chat/tts` 创建任务，再轮询 `/AI/chat/tts/query` 查询状态。虽然前端多了一步轮询，但服务端链路更符合第三方平台的真实运行方式。

### 7. 为什么还要加 Windows stub
这轮重构里有一个非常“工程而不炫技”的动作：新增了 `common/image/image_recognizer_windows.go`。

原因不是业务需要 Windows 线上运行，而是本地开发和验证环境需要兼容。当前用到的 `onnxruntime_go` 在 Windows 下并没有当前项目这套可直接工作的编译条件，如果完全不处理，会导致：

- 多模态相关包在本地 Windows Go 环境下无法编译
- 聚焦单测也没法跑
- 开发节奏被平台差异拖住

所以现在 Linux 正式实现放在 `//go:build !windows` 文件里，Windows 下走 stub，让包结构仍然可编译。这样既不影响生产链路，也让本地验证能继续推进。

这类“为研发环境减阻力”的改动，往往也是成熟项目和课堂 Demo 的分水岭。

## 四、为什么这一轮要这样设计，而不是继续堆接口
### 1. 因为多模态不是“多两个 controller”
如果图像识别和 TTS 只是两个零散接口，那它们确实只要 controller + service 就能先跑起来。但一旦继续往后扩展：

- 图片预处理可能会升级。
- 模型可能会更换。
- 语音平台可能会切换供应商。
- 后续可能加 OCR、ASR、语音克隆、视频理解。

这时候如果不先把“多模态运行时边界”立起来，后面会越来越乱。`internal/modules/multimodal` 的价值就在于先占住这个边界。

### 2. 因为运行时复用是性能问题，不是代码洁癖
很多人第一次看到“把 recognizer 做成单例”“缓存 access token”，会觉得这只是代码优化。其实不是。

它真正解决的是：

- 平均响应时间
- 峰值并发下的稳定性
- 第三方接口的调用成本
- 模型运行时的初始化开销

也就是说，这些动作表面上是重构，底层上是在修系统的性能模型。

### 3. 因为上传接口天然是风险边界
文件上传类接口一直是高风险区域。图像识别看起来只是“传一张图”，但本质上它仍然是文件进入后端系统的入口。

所以这一轮不是只关心“能识别”，而是同时关心：

- 能不能限制大小
- 能不能识别伪造文件
- 能不能避免异常输入把推理层搞崩
- 能不能把错误在 controller 层优雅转成用户可理解的响应

这就是工程思维和功能思维的区别。

## 五、实现中必须特别注意的坑
### 1. 不能只信 `multipart.FileHeader.Size`
`FileHeader.Size` 可以作为第一道拦截，但不能完全信任它。真正读文件时仍然要用 `io.LimitReader` 控制上限，避免超大文件被完整读进内存。

### 2. 不能只做后缀名判断
只看 `.jpg`、`.png` 不够，必须补 MIME 检测和实际解码。否则扩展名伪装文件很容易绕过。

### 3. 不能把第三方 access token 当普通配置常量
第三方 token 是有生命周期的运行时数据。没有缓存、没有刷新缓冲区、没有过期判断的实现，在真实流量里会非常吃亏。

### 4. 第三方请求必须带 `context`
这样用户取消请求、网关超时、上游断开时，内部 HTTP 调用也能及时结束，避免僵尸请求积压。

### 5. 共享推理器要同时考虑性能和正确性
只做共享、不做锁，会有数据竞争。只做锁、不做复用，又回到了每次重建的慢路径。真正好的实现要同时满足这两件事。

### 6. 本地开发环境和生产环境的依赖矩阵可能不同
这轮的 ONNX Runtime 就是一个典型例子。工程里不能假设“所有平台都能一样编译”，必要时要用 build tag 或 stub 给开发环境留出空间。

## 六、围绕本篇内容，需要掌握哪些基础知识
### 1. 什么是 ONNX，什么是 ONNX Runtime
ONNX 是一个模型表示格式，目标是让不同训练框架导出的模型可以在统一的推理运行时中执行。ONNX Runtime 则是负责真正加载模型、准备输入输出、执行推理的运行时。

这也是为什么项目里不是“直接拿一张图去算”，而是要先准备：

- 模型文件
- label 文件
- 输入尺寸
- 输入 tensor
- 输出 tensor

### 2. 图像分类里的预处理为什么重要
在这一版实现里，图片会经历：

1. 缩放到固定输入尺寸
2. 提取 RGB 通道
3. 归一化到 `0~1`
4. 按 `NCHW` 排列写进 tensor

这一步如果和模型训练时的预处理不一致，哪怕推理链路完全能跑，结果也会明显偏掉。也就是说，多模态模型接入并不是“能调用就行”，而是“预处理必须对齐训练假设”。

### 3. 为什么 `http.DetectContentType` 只是嗅探，不是终局安全方案
它适合做服务端第一轮内容识别，但并不是完整的文件安全扫描系统。它只能帮助我们更快判断“这看起来像不像图片”，最终仍然要结合解码、大小限制、目录隔离，甚至后续的对象存储策略一起看。

### 4. 什么是 access token 的生命周期管理
第三方平台的 access token 通常具备：

- 申请成本
- 有效期限
- 失效时间
- 刷新策略

工程里真正要做的不是“会拿 token”，而是“会管理 token 生命周期”。这和 JWT、Redis 黑名单、Session 失效控制，本质上都属于凭证生命周期治理问题。

### 5. 为什么异步 TTS 更像任务系统而不是函数调用
当 TTS 变成长文本、批量任务、第三方资源调度后，它的语义就不再是“输入文本，立刻返回音频”，而更像：

- 创建一个合成任务
- 获取任务编号
- 查询任务状态
- 成功后下载结果

如果理解了这一点，你就会知道为什么前端轮询不是“多此一举”，而是对第三方任务模型的自然适配。

## 八、提问 + 标准答案
下面这组问题会更贴近国内大中厂围绕多模态接入、第三方平台治理和工程稳定性的常见追问。

### 问题 1：为什么图像识别不能每次请求都重新加载 ONNX 模型？
**标准答案：**
因为模型加载和 ONNX Runtime session 初始化成本很高，如果每次请求都重新创建，会显著拉高延迟并放大 CPU、内存抖动。更合理的做法是复用长生命周期的推理器，把初始化成本摊平到多个请求上。

### 问题 2：既然做了全局复用，为什么还要给推理器加锁？
**标准答案：**
因为共享推理器内部持有 session、输入 tensor 和输出 tensor，这些对象在推理过程中会被修改。多个请求同时写入共享 tensor 会产生数据竞争，导致结果串台或 session 运行异常，所以必须用互斥锁保护关键临界区。

### 问题 3：为什么服务端不能只根据文件后缀判断是不是图片？
**标准答案：**
因为文件后缀很容易伪造，攻击者完全可以把任意文件改名成 `.jpg` 再上传。更稳妥的做法是同时检查扩展名、MIME 类型，并尝试解码图片头，形成多层校验。

### 问题 4：为什么 `http.DetectContentType` 还不够，还要再做 `image.DecodeConfig`？
**标准答案：**
因为 `DetectContentType` 只是基于前几个字节做嗅探，适合做初筛，但它不等于真正完成了格式解析。`image.DecodeConfig` 能进一步验证这份内容是否真能被图片解码器识别，能更有效地挡住伪造文件。

### 问题 5：为什么上传时要同时校验 `FileHeader.Size` 和实际读取长度？
**标准答案：**
因为元数据可以作为第一道快速拦截，但不能完全依赖它。真正读文件时仍然要用 `io.LimitReader` 控制最大读取量，避免超大文件被完整读入内存。

### 问题 6：为什么 TTS access token 需要缓存？
**标准答案：**
因为 access token 的申请本身是一次额外网络请求，如果每次创建任务和查询任务都重新申请，会放大延迟，也更容易触发第三方限流。缓存 token，并根据 `expires_in` 做过期控制，是接第三方平台的基本工程能力。

### 问题 7：为什么 token 缓存还要设置一个 buffer，而不是等精确过期再刷新？
**标准答案：**
因为真实系统里还要考虑网络延迟、时钟偏差和并发请求。如果等到最后一秒才刷新，很容易出现边界时刻的失败。提前预留 buffer 可以减少临界点抖动。

### 问题 8：为什么当前 TTS 接口设计成“创建任务 + 查询结果”，而不是一次接口直接返回音频？
**标准答案：**
因为当前接入的第三方平台本身就是异步任务模型。长文本合成往往需要更多资源调度，平台更适合先返回 task_id，再让客户端查询状态。系统设计应该适配上游服务的真实语义，而不是强行包装成同步函数。

### 问题 9：如果未来要支持长文本批量转语音，这一版设计还能扩展吗？
**标准答案：**
可以。因为现在已经把 TTS 调用收口到独立服务模块里，后续加任务持久化、队列削峰、回调通知、失败重试，都有明确挂点，不需要把逻辑散回 controller。

### 问题 10：为什么第三方 HTTP 请求一定要传 `context.Context`？
**标准答案：**
因为请求取消、超时和链路终止都应该往下游传播。如果不传 `context`，用户请求都结束了，第三方调用还在后台继续跑，会浪费连接和计算资源。

### 问题 11：为什么这一轮要把多模态参数做成配置，而不是直接写死在代码里？
**标准答案：**
因为模型路径、图片尺寸、上传大小、TTS 音色、语速、超时都属于运行时参数，不同环境和不同业务场景都可能变化。配置化之后，切环境、调参数、做灰度都会更轻松。

### 问题 12：为什么多模态能力不应该只理解成“多两个接口”？
**标准答案：**
因为它背后其实有运行时管理、模型复用、第三方凭证生命周期、上传安全、异步任务编排这些系统问题。真正复杂的不是 controller 数量，而是这些能力如何稳定地活在同一个项目里。

### 问题 13：为什么要单独建 `internal/modules/multimodal`，而不是继续把逻辑散在 `common` 和 `service` 里？
**标准答案：**
因为图像识别和 TTS 虽然技术实现不同，但都属于多模态输入输出能力。把它们收拢成一个业务模块，有助于统一边界、配置、扩展方向和后续博客文档，也更符合按业务能力组织项目的思路。

### 问题 14：为什么这次还专门做了 Windows stub？
**标准答案：**
因为开发环境不一定和生产环境完全一致。当前 ONNX Runtime 在这个项目的 Windows 编译环境下不可直接工作，如果不做平台兼容处理，本地开发和聚焦验证都会被拖住。用 build tag + stub 可以在不影响 Linux 正式运行的前提下，保留 Windows 端的可编译性。

### 问题 15：如果第三方 TTS 平台开始限流，你会优先怎么优化？
**标准答案：**
我会先做 token 缓存和请求复用，再补调用级限流、失败重试、指数退避和错误码分级。若业务量继续增大，再考虑把创建任务这一层异步化，接入队列做削峰。

### 问题 16：为什么图片识别服务要做成惰性初始化，而不是应用启动时立刻加载模型？
**标准答案：**
因为模型文件可能不是所有环境都具备，启动时强制加载会把一个可选能力变成强依赖。惰性初始化可以让服务先启动起来，等真正用到图像识别时再创建推理器，更适合当前阶段的模块演进方式。

### 问题 17：如果面试官问你“同步 TTS 和异步 TTS 的差别”，你会怎么答？
**标准答案：**
同步 TTS 更像一次普通函数调用，适合短文本和快速返回；异步 TTS 更像任务系统，适合长文本、资源重的场景，前端通常要处理任务状态和结果轮询。选哪种不是喜好问题，而是上游能力和业务场景决定的。

### 问题 18：如何理解这一轮的“性能优化”和“架构优化”其实是同一件事？
**标准答案：**
因为这次架构上的抽象，直接对应的是运行时行为变化：图像识别从每次重建变成全局复用，TTS 从每次取 token 变成带生命周期缓存。也就是说，模块化不是空洞分层，而是在改变系统的性能模型和稳定性模型。

### 问题 19：如果未来加入 OCR、ASR、视频理解，这一版还能支撑吗？
**标准答案：**
可以，因为现在已经先把多模态边界立住了。后续新增能力时，可以继续沿着 `internal/modules/multimodal` 往下拆子模块，而不是再回到散落 controller/service/common 的状态。

### 问题 20：如果让你继续优化这一块，下一步你会补什么？
**标准答案：**
我会优先补这些能力：

- 图像识别结果缓存和模型预热策略
- TTS 任务持久化与失败重试
- 多模态链路监控指标与耗时埋点
- 第三方错误码分级与告警
- 更细的上传安全策略
- 后续 OCR、ASR 接入的统一接口规范

这说明项目会从“多模态能跑”继续走向“多模态稳定可运维”。

## 九、本篇小结
这一篇真正完成的，不是“多接了图像识别和 TTS”，而是把 GopherAI 的多模态能力从零散功能提升成了正式模块：

- 多模态运行参数被统一配置化。
- 图像识别从每请求重建模型升级成了共享运行时。
- 共享 ONNX 推理器具备了并发保护。
- TTS 第三方 token 开始具备生命周期缓存能力。
- 图像上传链路补上了后缀、MIME、解码三层校验。
- Windows 本地开发和 Linux 正式运行的差异被 build tag 平滑处理。

如果你把这一篇吃透，后面继续做 OCR、ASR、视频理解、语音对话，就不再只是“往项目里接一个新接口”，而是在“一个有边界的多模态系统”上继续生长能力。
